# 报关模块完整实施方案（一期 · 进口）

> **目的**：在《[报关_移库](./报关_移库.md)》已定稿规则下，将「报关完成 + 境内移库过账」及配套能力从占位推进到**可上线闭环**，并列出依赖产品/运维的输入。  
> **不在本文重复**：方案 C、虚拟调拨 3、1:1:1、流水 `STOCK_TRANS` 等业务理由——以《报关_移库》为准。

## 1. 目标与一期范围

### 1.1 必须达成（闭环）

| 编号 | 能力 |
|------|------|
| G1 | `CompleteDeclarationAndTransferAsync` **去掉占位抛错**，在同一数据库事务内完成：校验 → 报关内部状态落「完成」→ 创建并确认 **`stocktransfer_customers` / `stocktransfer_item_customers`** → **逐行或批量**调用 **`IInternalTransferPostingKernel`**（`Kind=Customs`）→ 回写每行 **`TargetStockItemId`**；任一步失败整单回滚（与《报关_移库》§5 一致）。 |

| G3 | **多行报关明细**：每行有 **`SourceStockItemId`**、**`DeclareQty`**（见实体 `CustomsDeclarationItem`）；移库行 **`Qty`** 与之一致或可审计推导；每行一次内核调用或扩展为批量 API（见 §4）。 |

### 1.2 一期建议包含（与《报关_移库》§9 一期对齐）

- **出库通知 → 审批 → 生成报关壳单**（若业务强依赖全链路，则与本方案 **并行或前置** 排期；见 §6）。
- 全库库存相关列表/汇总对 **`STOCK_TRANS` / 调拨类型 3** 的排除规则（与《报关_移库》§7 + 共享内核 checklist）。
- 前端：报关单编辑/完成弹窗（双确认见 §8）、物流菜单；RBAC 与操作日志（与 §8）。

### 1.3 可分期（不阻塞 G1–G3）

- 附件多文件、费用试算 UI 与 EBS 第 3 章逐项对齐的复杂交互。
- 移库冲销、出口报关、普通 Internal 调拨（二期）。

## 2. 现状盘点（开发基线）

### 2.1 数据模型（已具备）

- **`customs_declaration`**：`StockOutRequestId`、`FromWarehouseId`、`ToWarehouseId`、`InternalStatus`、`CustomsClearanceStatus`、报关公司等（`CustomsDeclaration`）。
- **`customs_declaration_item`**：**`SourceStockItemId`**、`DeclareQty`、`StockOutRequestId`、`MaterialId`、费用与 HS 等快照字段（`CustomsDeclarationItem`）。  
  → **与在库行的绑定已在明细层落地**；争议点主要在「谁、何时写入这些字段」及与出库执行的时序（见 §6）。
- **`stocktransfer_customers` / `stocktransfer_item_customers`**：`SourceStockItemId`、`CustomsDeclarationItemId`、`StockOutRequestId`、`Qty`、`TargetStockItemId`（`StockTransfer` / `StockTransferItem`）。
- **`stockledger` / `InventoryLedger`**：已含 `from_warehouse_id`、`to_warehouse_id`、`customs_declaration_id`、`stock_transfer_id`、`source_stock_item_id` 等扩展列（以实际迁移为准）。

### 2.2 服务与 API（缺口）

- **`CustomsDeclarationService.CompleteDeclarationAndTransferAsync`**：校验后 **固定抛错**，未创建移库单、未调内核（占位）。
- **`InternalTransferPostingRequest`**：**仅支持单行**（`SourceStockItemId` + `MoveQty` + 单一 `TransferItemLineId`）；文档 §5.1/§11 中的 **`LineSnapshots` 批量** 在代码中 **不存在**。
- **`InternalTransferPostingKernel`**：已实现调拨虚拟出/入、ledger、`Kind` 影响 Remark；**手工**整行出清会写 `TransferType=10`**；**报关**应使用 `Kind=Customs` 且 **不写** `TransferType=10`（避免报关源行被全库列表误藏）。

### 2.3 API 控制器（现状）

- 报关单：列表、按 id、按出库通知、PATCH 海关状态、POST complete（进占位服务）、删除/强制删除。
- **缺**：报关单/明细的 **POST/PUT/PATCH 创建与维护**（若产品要求系统内录单而非仅导入）。

## 3. 总体架构与事务边界

```
[Controller] POST .../complete
    → [Orchestration 编排层] CustomsDeclarationService（或独立 CustomsDeclarationTransferOrchestrator）
          1) 加载报关单+明细+仓库地域校验
          2) 状态/海关已结关/无重复移库单
          3) BEGIN 事务（EF 同一 DbContext / IUnitOfWork）
          4) 生成移库单头+多行（TargetStockItemId 空）
          5) FOR EACH line OR BatchKernel:
                 调 IInternalTransferPostingKernel.ExecuteAsync(...)
                 回写 StockTransferItem.TargetStockItemId
          6) 更新报关单 InternalStatus=Completed，写审计
          7) COMMIT
```

- **禁止**：在编排层手写 SQL 直接改 `stock`/`stock_item`（《报关_移库》§5.1）。
- **库存数量唯一写口**：`InternalTransferPostingKernel`。

## 4. 核心设计决策：多行如何调内核

文档要求 **「多行则 LineSnapshots 一批」**；当前 **无该 DTO**。**一期已拍板：采用方案 A（循环）**，见 §11.1。

### 方案 A（一期定稿）：编排层 **循环** 调现有 `ExecuteAsync`，**同一事务**

- **做法**：对每个 `StockTransferItem`（或按 `CustomsDeclarationItem` 排序后的行）构造一次 `InternalTransferPostingRequest`：  
  - `Kind = InternalTransferKind.Customs`  
  - `MoveQty = line.Qty`（与 `DeclareQty` 一致，见 §5.2）  
  - `TransferHeaderId = StockTransfer.Id`（与虚拟凭证 `SourceId` 对齐）  
  - `TransferBusinessCode = StockTransfer.TransferCode`  
  - **`TransferItemLineId = StockTransferItem.Id`**（满足 ledger `BizLineId` 与行 1:1，且与《报关_移库》§6.1 `BizLineId`=移库行主键一致）  
  - `FromWarehouseId` / `ToWarehouseId` = 报关头或移库头字段（须与源行仓库、目标仓一致）  
  - `LedgerBizType = StockLedgerBizType.StockTransfer`  
  - `ToLocationId`：一期可 null，内核回落为源行 `LocationId`（与手工一致）。
- **事务**：整段包在 **同一 `SaveChanges`/UnitOfWork 事务**；内核内若已 `SaveChanges`，需统一为 **「编排只调不中途提交」** 或 **「内核不 Save、由编排统一 Save」**——**必须核对** `IUnitOfWork` 与内核内 `SaveChangesAsync` 调用，避免半提交；必要时给内核增加 `skipSave` 或抽出 **纯领域方法 + 编排统一持久化**（实现任务子项）。
- **部分失败**：任一 `ExecuteAsync` 抛错 → 整单回滚；报关单状态回到 **报关中**（《报关_移库》§5.2）。
- **幂等**：仍以「不存在已成功移库单 + 报关未完成」为门槛；重复点击 complete 应 **400 明确提示**。

### 方案 B（后续优化）：扩展 `InternalTransferPostingRequest` 增加 `IReadOnlyList<LineSnapshot>` **单次**内核

- 优点：单次循环内序列号/日志更集中；减少多次 `SaveChanges` 风险。  
- 缺点：内核改动面大、测试矩阵陡增。  
- **一期已选方案 A**；`LineSnapshots` 批量内核为 **二期或重构项**（见 §12）。

## 5. 编排层详细步骤（建议伪代码级任务拆分）

### 5.1 前置校验（除现有外建议补充）

| 校验项 | 说明 |
|--------|------|
| 明细非空 | `Items` 至少一行且未软删。 |
| 源行存在 | `SourceStockItemId` 可加载；`WarehouseId` 与 `FromWarehouseId`（或报关头约定）一致。 |
| 地域 | 源仓 **境外**、目标仓 **境内**（与 `WarehouseInfo.RegionType` 及《报关_移库》§2.1 一致）。 |
| 数量 | `DeclareQty > 0` 且 `DeclareQty <=` 对应 `StockItem.QtyRepertoryAvailable`（或业务确认用 `QtyRepertory`）；**报关行与源行物料/桶键**一致性抽检（`MaterialId` / PN+Brand 与 `StockItem`）。 |
| 汇总桶 | 目标侧分桶与内核一致（内核已 `StockMatchesInboundBucket`）；**多行不得要求同一目标 Stock 桶**——每行独立解析目标桶。 |
| 拣货阻塞 | 复用手工移库内核前对源行的 **未完成拣货任务** 校验（内核内已有 `GetPickingBlockReasonsAsync`）。 |

### 5.2 数量口径（已拍板）

- **`DeclareQty` 即过账数量**；生成移库行时 **`StockTransferItem.Qty` 与 `DeclareQty` 一致**；内核 `MoveQty` 取同一值。  
- 不再拆「申报数量 / 过账数量」第二字段。

### 5.3 移库单写入顺序（建议）

1. `SerialNumber` 生成 `TransferCode`（`ModuleCodes.StockTransfer` / `STF`）。  
2. `StockTransfer`：`BizScene=CustomsImport`，`CustomsDeclarationId`，`From/To`，`Status=Confirmed`，`ConfirmedTime/By`。  
3. 为每行报关明细建 `StockTransferItem`：`Qty`、`SourceStockItemId`、`CustomsDeclarationItemId`、`StockOutRequestId`。  
4. 循环调内核，**每行成功**后 `item.TargetStockItemId = result.TargetStockItemId`（内存更新后一次 Save 或按单元工作单元策略）。  
5. `CustomsDeclaration.InternalStatus = Completed`；`ModifyTime`/`ModifyByUserId`。  
6. 触发与库存相关的 **反范式同步**（如备货 `PurchasedStock_AvailableQty`）：与手工移库成功后一致，抽取公共方法或复用现有 `TryRecalculateFromChangedStockInfosAsync`（按变更的 `StockInfo` 列表）。

### 5.4 与 `TransferType`（手工移库源行隐藏）的关系

- **报关迁库**：源行使用 `Kind=Customs`，**不应**打 `TransferType=10`（该标记仅用于手工移库列表过滤）。  
- 若报关后也希望在列表隐藏「已迁空的境外源行」，须 **单独常量/产品决策**，避免与手工语义混淆。

## 6. 前置链路：出库通知 → 审批 → 报关壳（§2）

《报关_移库》§2.2 写明：**出库创建时**判断地域不一致 → **待审批**；**通过后**生成待报关/报关壳；**与在库行绑定**的细节依赖「实现阶段 / PRD 细则」。

### 6.1 已拍板（与 §11 一致）

| 项 | 结论 |
|----|------|
| **一期建单方式** | **接受手工**：报关头/明细由用户在系统内 **新建与维护**（不做「审批通过即自动生成壳单」的强制依赖）；**报关完成 + 移库过账**（§5）以「壳单与明细已存在且合法」为前提即可开发。 |
| **`SourceStockItemId`（自动链路目标态）** | **模式 A**：当后续实现「审批通过 → 系统自动生成待报关壳单」时，在 **生成壳单环节** 由系统按规则 **自动带出** 每行 `SourceStockItemId`（与出库/拣货推荐或 FIFO 等规则在开发任务中细化）。 |
| **与一期手工的关系** | **不矛盾**：一期无自动壳单时，由关务在 **报关编辑页** 维护/选择 `SourceStockItemId`；自动壳单上线后，生成瞬间用模式 A 写入，仍可保留人工改行（若后续 PRD 要求）。 |

**仓库与头表（与 §5 校验一致）**：每一行 `SourceStockItemId` 对应 `stock_item.WarehouseId` **必须等于** 报关头 `FromWarehouseId`；境内入账目标仓 **必须等于** 报关头 `ToWarehouseId`。

### 6.2 开发任务建议

- 子项目 **SOR-CUSTOMS-01**（可二期）：在 `StockOutRequest` 创建/更新路径插入地域判断 + 写审批。  
- 子项目 **SOR-CUSTOMS-02**（可二期）：审批通过监听/轮询任务 → 生成 `customs_declaration` + `items`（内部状态 `Pending`）+ 生成 `DeclarationCode`；明细行 **`SourceStockItemId` 按 §6.1 模式 A** 自动带出。  
- **一期优先**：报关单/明细 **CRUD + 校验** + **§5 报关完成过账**；审批自动生成壳单与 SOR-CUSTOMS-* **可后置**。

## 9. 测试与验收

### 9.1 场景矩阵（最低）

| 场景 | 预期 |
|------|------|
| 单行、整行迁完 | 源 `QtyRepertory` 正确；目标仓新 `StockItem`；虚拟出/入类型 3；ledger 一条；移库行 `TargetStockItemId` 非空；报关完成。 |
| 多行、不同源行 | 每行独立目标桶；无双扣；事务全成或全败。 |
| 第二行失败 | 第一行回滚（若单事务）；报关状态仍为报关中。 |
| 重复 complete | 第二次 400；无重复移库单与流水。 |
| 海关未结关 | 400。 |
| 拣货占用阻塞 | 与手工移库一致提示。 |

### 9.2 数据对账

- `stocktransfer_item_customers` 行数 = 报关明细行数；`SUM(Qty)` 与报关申报数量合计可核对。  
- `stockledger`：`BizType=STOCK_TRANS`，`BizId`=移库单 Id，`BizLineId`=移库行 Id。

## 11. 为完成本方案，需要你（或产品）配合的事项

### 11.1 必须决策（已书面拍板 · 2026）

以下五项已由需求方确认，开发按此实现；与《报关_移库》冲突时以 **本文 §11 + §5/§6** 为实施口径。

| # | 议题 | 拍板结论 |
|---|------|----------|
| 1 | 多行调内核：§4 **方案 A（循环）** vs 方案 B（批量内核） | **方案 A**：多行时 **编排层循环** 调用 `IInternalTransferPostingKernel.ExecuteAsync`，**同一事务**内完成；**`LineSnapshots` 批量内核列为二期优化**（见 §12）。 |
| 2 | `DeclareQty` 是否即过账数量 | **是**。`DeclareQty` 即过账数量；**`stocktransfer_item_customers.Qty` 与 `DeclareQty` 一致**；内核 `MoveQty` 同值。 |
| 3 | 头表 `FromWarehouseId`/`ToWarehouseId` 与每行源仓 | **最严方案 A**：过账时每一行 `SourceStockItemId` 的 `stock_item.WarehouseId` **必须等于** 报关头 `FromWarehouseId`；境内入账目标仓 **必须等于** 报关头 `ToWarehouseId`。 |
| 4 | `SourceStockItemId` 由谁、何环节写入 | **模式 A（目标态）**：**审批通过并系统生成待报关壳单时**，由系统 **自动带出** 每行 `SourceStockItemId`（具体算法在 SOR-CUSTOMS-02 任务中细化）。 |
| 5 | 审批挂接 + 一期是否仅手工壳单 | **一期接受手工**：报关头/明细 **人工维护**，**不强制** 依赖审批自动生成壳单；审批 + 自动壳单（含上表第 4 点）**可二期接入**，与一期手工建单 **不矛盾**（见 §6.1）。 |

### 11.2 强烈建议提供（减少返工）

6. **2～3 组真实或脱敏的联调用例**：出库通知 id、境外仓/境内仓、多行明细、期望迁前后库存快照。  
7. **验收签字人**与 **上线窗口**（是否允许分阶段：先 internal 环境仅完成 G1）。  
8. **附件与费用试算**：一期做还是二期；若一期做，存储介质与字段清单。

### 11.3 环境与安全

9. 测试库 **迁移执行顺序** 与生产 **回滚预案**（报关完成一般不允许业务回滚，仅技术故障回滚）。  
10. 是否需要 **关务角色** 与 **仓库角色** 分离（RBAC 命名）。
