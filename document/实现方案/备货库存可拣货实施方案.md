# 备货库存可拣货 — 业务逻辑与实施方案

> 本文档说明：**在拣货环节**如何把 **备货类型（`StockType = 2`）** 且在业务上**允许用于本销售出库**的在库明细纳入**人工可拣候选**，并与《库存拣货出库整改方案》中「**任务壳 + 仅 `stockitem`、候选列表 + 人工确认**」衔接。  
> **申请出库**阶段的 `availableStockQty` 是否纳入备货池，可按产品分阶段推进（见文末）。

---

## 一、业务背景

### 1.1 当前与目标差异（概念层）

| 维度 | 当前常见行为 | 目标（拣货侧） |
|------|----------------|----------------|
| 客单库存 | `stockitem` 带 `sell_order_item_id`，与销售行强绑定，易进入候选。 | 继续作为候选主力。 |
| 备货库存 | 可能 **无** `sell_order_item_id`，或 `MaterialId` 形态与销行键不一致；若候选接口只按销售行 Id 过滤，**不会**出现在待拣货列表。 | 在 **PN/品牌等业务规则** 满足时，**进入同一待拣货候选列表**（FIFO 仅排序），由物流**人工勾选**。 |

### 1.2 与整改方案的关系

- 整改方案已定：**拣货与执行以 `stockitem` 行为准**，`pickingtaskitem` 存 `stock_item_id` + `stock_id`。  
- **备货可拣货** 的实现重心在：**「查询待拣货候选 `stockitem`」接口的筛选与排序规则**，而不是在执行出库时再区分客单/备货。

---

## 二、业务逻辑

### 2.1 备货库存的识别

- 汇总层 **`stock.StockType = 2`** 表示备货（与采购订单头类型、常量定义一致）。  
- 在库明细 **`stockitem`** 继承桶的物料与关联字段；候选列表以 **`stockitem` 行为单位** 返回时，需能判定其所属 **`stock`** 的 `StockType`（或通过关联 `StockAggregateId` 查汇总行）。

### 2.2 何时允许备货进入「可拣候选」

在**不放宽到全仓乱拣**的前提下，建议同时满足：

1. **仓库一致**：`stockitem.WarehouseId` = 当前拣货任务/出库通知所选仓库。  
2. **可用量**：`QtyRepertoryAvailable > 0`。  
3. **业务匹配（备货专用）**：与销行在 **型号 PN + 品牌** 上可比对一致，规则与现有工具类对齐思路：  
   - 参考 `CRM.Core.Utilities.StockMaterialMatch` 中 **`StockingSupplementEligible` / `StockingMatchesSellOrderForPicking`** 的语义：  
     - 优先用 **物料主数据** 解析库存行对应型号/品牌，与 **销售订单明细** `PN`、`Brand` 比对（忽略大小写）；  
     - 无主数据时，可用库存关联 **采购明细行** 的 PN/品牌与销行比对（与注释「与库存中心展示口径一致」对齐）。  
4. **销售行上下文**：候选接口必须传入 **出库通知对应的 `sell_order_item_id`（及订单头）**，以便做上述比对。

**不满足 3 的备货 `stockitem`**：不出现在候选列表（或单独分组「不可用」仅只读展示，由产品决定；默认不展示更简单）。

### 2.3 FIFO 与人工选择

- 备货与客单 **同一套规则**：候选列表 **FIFO 仅排序**；**是否拣备货行**由人工勾选。  
- UI 建议对行打 **来源标签**：如「客单库存」「备货库存」，避免误操作（与现有前端展示思路可复用）。

### 2.4 数量与冻结

- 与总方案一致：**已选行数量之和 = 出库通知数量** 才允许「完成拣货」；完成后**不可改**。  
- 备货行与客单行 **无优先级强制**：仅排序建议；**超额、错物料**由保存前服务端校验拦截。

### 2.5 与执行出库

- 一旦写入 `pickingtaskitem`（含备货对应的 `stock_item_id`），**执行出库不区分**客单/备货，**同一扣减逻辑**；溯源靠 `picking_task_item_id`。  
- **扩展表 / 利润快照**：若备货行价格字段与客单行不同，沿用现有 `BuildStockOutItemExtend` 等路径按实际 `stockitem` 取数即可。

## 三、实施方案

### 3.1 候选 `stockitem` 查询（后端）

**输入（示例）**：`stock_out_request_id` 或 `sell_order_item_id` + `warehouse_id`（+ 可选物料键用于二次过滤）。

**查询逻辑（建议两段 UNION 或两次查询合并去重）：**

**集合 A — 客单强绑定（现有思路）**

- `stockitem.sell_order_item_id` = 当前销售明细 Id；  
- 本仓、`QtyRepertoryAvailable > 0`；  
- 所属 `stock` 与通知物料/业务规则一致（与现网校验对齐）。

**集合 B — 备货补充**

- `stock.StockType = 2`（且 `stockitem` 归属该桶）；  
- 本仓、`QtyRepertoryAvailable > 0`；  
- **`StockMaterialMatch.StockingSupplementEligible`**（或等价拆分步骤）为 **true**：即备货 + 销行 PN/品牌与库存解析结果一致。

**合并后**

- 去重（同一 `stockitem` 只一行）；  
- **FIFO 排序**：`ProductionDate ?? CreateTime`，再 `CreateTime`；  
- 返回 DTO：含 `stockItemId`、`stockAggregateId`、`availableQty`、`stockType`、展示用 PN/品牌/库位/批次等。

### 3.2 保存拣货结果

- 用户勾选后可包含 **集合 A 与集合 B** 的任意组合；  
- 每行仍写入 **`pickingtaskitem.stock_item_id` + `stock_id`**；可选增加 **`IsStockingSupplement`** 冗余标记（若表已有该字段可沿用，便于报表）。

### 3.3 权限与审计（建议）

- 角色：是否仅 **仓管 / 物流** 可保存拣货、是否允许勾选备货，可用 RBAC 控制（按你们现有权限体系挂菜单/接口权限）。  
- 日志：保存与完成拣货记操作人、请求体摘要，备审计。

### 3.4 测试要点

- 仅有客单候选、仅有备货候选、**混合**候选三种列表；  
- PN/品牌边界：销行缺 PN/品牌时是否与 `StockMaterialMatch` 一致（通常应 **不匹配** 备货，避免整池暴露）；  
- 可用量为 0 的备货行不出列表；  
- 完成拣货后执行出库，**备货行**库存扣减与 `stockoutitem` 溯源正确。

## 四、与「申请出库」阶段的关系（分阶段说明）

当前 **`GetAvailableQtyForSellOrderItemAsync`** 仅统计 **`sell_order_item_id` 精确匹配** 的可用量，**不包含**仅靠 PN/品牌匹配的备货池（历史对话已确认）。

| 阶段 | 是否纳入备货 | 说明 |
|------|----------------|------|
| **拣货候选（本方案）** | **是**（按上节规则） | 与「人工在库位上拣备货」一致。 |
| **申请出库 / 弹窗「在库可用」** | **可选二阶段** | 若产品要求与拣货一致，可后续改为「客单匹配 + 备货池（同 PN/品牌）」汇总，需单独改 `StockOutService.GetApplyContextAsync` 与前端提示文案，避免与拣货可拣范围不一致。 |

**建议**：首版先完成 **拣货候选 + 执行出库链路**；若业务反馈「申请阶段在库可用仍为 0 但仓库明明有备货」，再立项对齐 **`availableStockQty`** 口径。

## 五、风险与依赖

- **物料主数据质量**：备货匹配依赖 `MaterialInfo` 或采购行 PN/品牌；主数据缺失时匹配变窄。  
- **同 PN/品牌多销售行**：备货池可能被子午线共享（与 `SellOrderItemExtend.PurchasedStock_AvailableQty` 文档一致）；拣货仍应按 **当前通知的销售行** 做展示与校验，**扣减仍按实物 `stockitem`**，不与「额度快照」混为一谈。  
- **管理员异常**：与总方案一致，误拣/误完成由管理员处理；后续可加产品化「解锁」流程。
