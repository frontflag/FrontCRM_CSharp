# 登录日志功能 — 实施方案（修订版）

**目标：** 记录用户成功登录的**日期时间**与基于 **ip2region** 的 **IP → 地理文字**；提供「系统日志」下「登录日志」菜单与列表页（搜索 + 表格），并与现有操作日志权限模型一致。

**页面规范：** 符合 [业务页面规范](../PRD/规范/UI规范/业务页面规范.md)，列表与搜索栏对齐 [业务列表规范](../PRD/规范/UI规范/业务列表规范.md)、[列表搜索栏规范](../PRD/规范/UI规范/列表搜索栏规范.md)，实现上对标 `CRM.Web/src/views/System/OperationLogList.vue`（`page-header`、`search-bar`、`CrmDataTable`、底部分页）。

**运行时数据：** `ip2region_v4.xdb` 等放在仓库根 `data/ip2region/`（已由 `.gitignore` 排除、构建脚本不打包）；详见根目录 `.gitignore` 与 `build_with_temp_path.ps1` 说明。

## 一、功能范围

| 项目 | 说明 |
|------|------|
| **记录内容** | **登录时间**（服务端写入时刻）、**登录人**（用户 Id + 展示名快照）、**客户端 IP**、**登录地址**（由 ip2region 解析并规范为「国家省市区街道」展示串）、**登录方式**（见下文枚举）。 |
| **地理解析** | **仅使用 ip2region** 离线库（`ip2region_v4.xdb`，路径可配置）；**不依赖**外网 HTTP 地理接口。IPv6 若需支持可二期增加 `ip2region_v6.xdb` 与双库查询逻辑。 |
| **记录时机** | **账号密码登录成功**：`POST /api/v1/auth/login` 在校验通过、即将返回 Token 前写入一条。**模拟登录成功**：`POST /api/v1/auth/impersonate` 成功写入一条，并记录操作者。**微信登录等**：当前不在范围；表结构预留 `login_method` 枚举值，成功路径统一调用 `ILoginLogService.RecordAsync(...)` 便于后续扩展。 |
| **失败登录** | **一期不记录**（可选二期：独立策略与限流，防刷库）。 |
| **权限** | 与操作日志一致：列表与查询 API 使用 **`rbac.manage`**（与 `OperationLogsController` 一致）。 |
| **隐私** | IP 与解析地址属敏感信息；访问角色与保留周期由运维/产品约定，可在系统管理 PRD 中补充。 |

### 1.1 登录方式（`login_method`）— **枚举存库**

**原则：** 库中存 **枚举（整型或 PostgreSQL ENUM）**，禁止用可变长字符串表示业务码；API 对外可用 **数字或字符串名**（与前端约定一致），由 DTO / JSON 序列化映射。

**取值表（与 C# `LoginMethod` 一致，推荐 `smallint`）：**

| 存库值（`smallint`） | 枚举名（C# / API 可选名） | 含义 | 一期 |
|----------------------|---------------------------|------|------|
| `1` | `Password` | 账号密码登录 | ✓ |
| `2` | `Impersonate` | 模拟登录（管理员代进员工账号） | ✓ |
| `3` | `Wechat` | 微信登录 | 预留，接入后写入 `3` |

**数据库二选一（实现时选定一种即可）：**

1. **`smallint NOT NULL`** + `CHECK (login_method IN (1, 2, 3))`（或 `1..3` 范围约束）— 与 EF Core `enum : short` 映射简单，迁移与跨工具友好。  
2. **PostgreSQL `CREATE TYPE ... AS ENUM ('password', 'impersonate', 'wechat')`**，列类型为该 ENUM — 语义直观；需在 Npgsql/EF 中注册映射或使用原始 ADO 读标签。

**推荐一期：** `smallint` + 文档化上表；C# 定义：

```csharp
public enum LoginMethod : short
{
    Password = 1,
    Impersonate = 2,
    Wechat = 3,
}
```

列表「登录方式」列：根据枚举值映射中文（账号密码登录 / 模拟登录 / 微信登录）；未知值显示 `—` 或原始数字并打日志。

## 二、数据库：`log_login`

**表名：** `public.log_login`。

**字段建议：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `uuid` PK | 主键 |
| `user_id` | `varchar(36)` 等 | 登录用户 Id（与 `sys_user` 一致） |
| `user_name` | `varchar` | 登录时用户名快照（列表「登录人」） |
| `login_at` | `timestamptz` | 登录时间；列表展示 **精确到分钟**（前端 `YYYY-MM-DD HH:mm`，后端可按秒存） |
| `client_ip` | `varchar(45)` | 列表「登录 IP」 |
| `country` | `varchar` nullable | 解析：国家 |
| `province` | `varchar` nullable | 省/直辖市等 |
| `city` | `varchar` nullable | 市 |
| `district` | `varchar` nullable | 区/县 |
| `street` | `varchar` nullable | **街道**：当前 ip2region 数据段多为 `国家|区域|城市|ISP|...`，通常 **无独立街道字段**，一期可恒为 `NULL`；若数据格式升级有街道再填。 |
| `address_line` | `varchar(500)` nullable | **列表「登录地址」主展示**：将 `country/province/city/district/street` 非空部分按「国家省市区街道」顺序拼接为单行；解析失败可为 `NULL` 或 `—` 由前端处理。 |
| `region_raw` | `varchar` nullable | ip2region **原始返回串**（便于排错与格式变更后重算） |
| `login_method` | **`smallint` NOT NULL**（或 PG **ENUM** 类型，见 §1.1） | 取值 `1`/`2`/`3`，与 `LoginMethod` 枚举一致 |
| `actor_user_id` | `varchar` nullable | 仅 `impersonate` 时填**管理员**用户 Id |
| `geo_source` | `varchar(32)` | 固定 `ip2region`；未解析或库缺失时 `none` |

**说明：** 当前需求列表**不包含「经纬度」列**；表结构可不建 `latitude/longitude`；若后续地图需求再增加字段即可。

**索引：**

- `idx_log_login_login_at` on `login_at DESC`
- `idx_log_login_user_id` on `user_id`

**交付物：** PostgreSQL 脚本（`scripts/add_log_login_*.sql`）或与现有 EF 迁移流程二选一对齐。

## 三、ip2region 集成要点

| 项目 | 说明 |
|------|------|
| **数据文件** | 使用官方 **`ip2region_v4.xdb`**（仓库外或 `data/ip2region/`，已 `.gitignore`）；配置项如 `Ip2Region:Ipv4XdbPath`（相对 `ContentRootPath` 或绝对路径）。 |
| **解析库** | 选用维护中的 **C# binding**（与 `ip2region` v3 xdb 兼容，以 NuGet 为准），启动时或首次查询加载 Searcher（单例、线程安全以库文档为准）。 |
| **IP 获取** | 从 `HttpContext` 取客户端 IP：**`X-Forwarded-For`** 最左非空 token → **`X-Real-IP`** → **`RemoteIpAddress`**（与反向代理部署一致）。 |
| **文本 → 结构化** | 对返回字符串按 **`|`** 分段；按当前数据格式映射到 `country/province/city` 等（注意直辖市、海外英文段与 `ISP` 列位置；与官方 README 中格式说明保持一致）。`address_line` = 业务约定顺序拼接（国家→省→市→区→街道），无则跳过。 |
| **失败降级** | 内网/保留段/库未加载：不写异常栈到表；`geo_source=none`，`address_line` 为空，**仍记录** `client_ip` 与 `login_at`。 |
| **性能** | 同步查询应 **毫秒级**；若担心偶发磁盘问题，可 try/catch 吞掉解析异常不影响登录主流程。 |

## 四、后端实现要点

| 模块 | 说明 |
|------|------|
| **写入** | `AuthController` 在 **`Login`**、**`Impersonate`** 成功返回前调用 `ILoginLogService.RecordAsync(...)`，传入：`userId`、`userName`、**`LoginMethod` 枚举**（`Password` / `Impersonate`）、`actorUserId?`、**`HttpContext`** 或已解析的 `IP`。不在无 `HttpContext` 的 `AuthService` 内取 IP。 |
| **服务** | `IIp2RegionLookup`（或内嵌于 `ILoginLogService`）：`TryLookup(string ip, out RegionParts parts)`；`ILoginLogService` 组装实体并 `SaveChanges`。 |
| **查询 API** | 新建 `LoginLogsController`，路由建议 **`GET /api/v1/login-logs`**，`[Authorize]` + `[RequirePermission("rbac.manage")]`。Query：`loginAtFrom`、`loginAtTo`（日期闭区间语义在接口文档写清）、`userId?`、`page`、`pageSize`。返回 DTO 含列表所需字段。 |
| **员工筛选** | 与 RBAC/用户列表已有接口对齐：下拉或远程搜索返回 `userId` + 显示名，请求登录日志时传 `userId`。 |

**参考代码：** `CRM.API/Controllers/OperationLogsController.cs`、`CRM.API/Controllers/AuthController.cs`、`CRM.API/Services/Implementations/AuthService.cs`。

## 七、测试与验收

| 场景 | 预期 |
|------|------|
| 密码登录成功 | `log_login` 一条；`login_method = 1`（`Password`）；`client_ip`、`address_line`/`region_raw` 合理；`login_at` 正确。 |
| 模拟登录成功 | `login_method = 2`（`Impersonate`）；`actor_user_id` 为管理员；`user_id` 为目标员工。 |
| 反向代理 | `X-Forwarded-For` 为真实客户端 IP。 |
| ip2region 缺失 | 登录仍成功；该条 `geo_source=none`，地址为空。 |
| 列表 API | 非管理员 403；日期区间 + 员工筛选 + 分页正确。 |
| 前端 | 菜单在操作日志之上；五列展示与需求一致；登录方式中文正确。 |
