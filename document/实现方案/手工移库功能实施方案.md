# 手工移库功能 — 实施方案（共享内核 + 虚拟出/入库）

> **文档定位**：本文件以 **手工移库** 为范围，约定需求、路由、API、前端、阶段验收；**库存过账与虚拟出入库** 与 **报关移库** 共用同一套 **共享内核**（详见《[移库共享内核_虚拟出入库实施方案](./移库共享内核_虚拟出入库实施方案.md)》）。  
> **硬约束**：**严禁双扣**（库存数量仅在内核变更一次）；**严禁**将调拨类虚拟出入库（`StockInType=3` / `StockOutType=3`）计入真实购销出入库统计或执行业务出库流程。

**版本**：v7 — 全文重写：对齐共享内核与对称虚拟凭证方案。

---

## 〇、与报关移库的关系

| 结论 | 说明 |
|------|------|
| **业务分离** | **报关移库**：前端 `/inventory/transfers-customers`，表 `stocktransfer_customers` / `stocktransfer_item_customers`，强绑 `customs_declaration_id`。 **手工移库**：前端 `/inventory/transfers-manual`，表 `stocktransfer_manual` / `stocktransfer_item_manual`，与报关无耦合。 |
| **过账复用** | 两者均通过 **`IInternalTransferPostingKernel.ExecuteAsync`** 完成：**物理扣源 + 物理加目标 + 虚拟出库 + 虚拟入库 + `stockledger`**；差异仅在于 **编排层** 的前置校验与主表写入。 |
| **虚拟凭证对称** | 手工与报关在落库后均应具备 **对称的 `stock_out`（调拨出库）+ `stock_in`（调拨入库）** 凭证，便于审计与对账；**不得**再走 `StockOutService`/`StockInService` 的正式过账入口二次改库存。 |

## 一、需求边界（一期）

| 项目 | 定义 |
|------|------|
| **数量** | **不提供**「移库数量」输入框；**不允许**用户部分数量移库。`MoveQty` 由编排层按源行 **`QtyRepertoryAvailable`**（或评审结论）计算，与现 `PreviewAsync` 一致。 |
| **虚拟出/入库** | 每条成功移库产生 **`StockOutType=3`（调拨出库）** 与 **`StockInType=3`（调拨入库）** 各一套主从表；`SourceId` 指向 **`stocktransfer_manual.Id`**，`SourceCode` 为手工移库单号（`TransferCode`）。 |
| **报关** | 手工移库 **不** 读写报关表、**不** 调用报关移库 API。 |
| **目标** | 必选 **目标仓库**；可选 **目标库位**（`LocationId` 与现网一致）。 |
| **经济字段** | 目标侧在库行 **继承** 源行采购价、币别、采销关联等（迁库不改价）。 |
| **RegionType** | 目标行按 **目标仓档案地域** 赋值；未维护则沿用源行并记录日志（可配置为硬错误）。 |
| **权限与审计** | 权限码 `inventory.manual-transfer.execute`（可评审）；重要操作写 **`log_operation`**（与项目统一操作日志一致）。 |

### 1.1 产品形态

- **形态 A（一期必做）**：按单条 `stock_item` **整行可迁量** 移库（与现实现一致）。  
- **形态 B（可选/二期）**：整仓批量迁 — 见共享内核文档多行 `LineSnapshots` 与异步策略。

## 二、数据模型

| 表 | 用途 |
|----|------|
| `stocktransfer_manual` | 手工移库主单：`TransferCode`、`FromWarehouseId`、`ToWarehouseId`、`Status`、`Remark`、`Confirmed*` 等。 |
| `stocktransfer_item_manual` | 明细：`SourceStockItemId`、`TargetStockItemId`、`Qty`（**执行快照**，非用户录入）。 |
| `stock_out` / `stock_out_item` | **虚拟调拨出库**（类型 3），凭证；**不驱动**二次库存扣减。 |
| `stock_in` / `stock_in_item` | **虚拟调拨入库**（类型 3），凭证；目标 `stock_item` 挂 `stock_in_item_id`。 |
| `stockledger` | 库存轨迹；`BizType` 使用 **`StockLedgerBizType.ManualTransfer`**（与财务评审一致）。 |

**与初版差异**：在引入内核前，代码曾仅写虚拟入库；**目标态**为内核内 **先物理账、后对称虚拟凭证**（顺序见共享内核文档 §四）。

## 三、架构与职责划分

```
ManualStockTransferService（编排）
  ├─ PreviewAsync / 拣货占用校验 / 同仓同库位校验
  ├─ 生成 TransferCode、写 stocktransfer_manual*（头与明细可先写或后写，须与内核约定 TransferHeaderId）
  └─ 调用 IInternalTransferPostingKernel.ExecuteAsync(InternalTransferPostingRequest)

InternalTransferPostingKernel（内核）
  └─ 唯一 stock/stock_item 数量写口 + 虚拟出 + 虚拟入 + ledger + 反范式同步
```

- **编排层** 禁止直接 `UPDATE stock`/`stock_item` 数量（除迁移过渡期外应全部迁入内核）。  
- **内核** 禁止解析「报关单」「手工单」业务名；仅消费 `Request` 中的 ID、数量、仓库、枚举 `Kind=Manual`、Ledger 类型等。

`InternalTransferPostingRequest` 字段与事务步骤以 **《移库共享内核_虚拟出入库实施方案》** 为准；手工移库固定：

- `Kind = Manual`  
- `LedgerBizType = ManualTransfer`（常量名以代码为准）  
- `TransferHeaderId` = `stocktransfer_manual.Id`  
- `TransferBusinessCode` = `TransferCode`

## 四、API 设计（手工）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/v1/inventory/manual-transfers/preview?sourceStockItemId=` | 预览可迁量、拦截原因；**不改库**。 |
| `POST` | `/api/v1/inventory/manual-transfers` | 执行；Body **无 `qty`**：`sourceStockItemId`、`toWarehouseId`、`toLocationId?`、`remark?`。 |
| `GET` | `/api/v1/inventory/manual-transfers` | 分页列表（P1）。 |
| `GET` | `/api/v1/inventory/manual-transfers/{id}` | 详情（P1）。 |

**权限**：`inventory.manual-transfer.execute` + `[Authorize]` 与 RBAC 一致。

**Controller**：`ManualStockTransfersController`，路由 `api/v1/inventory/manual-transfers`（已实现则保持）。

## 五、前端设计

| 项 | 说明 |
|----|------|
| **路由** | `/inventory/transfers-manual`（`InventoryTransfer.vue`）。 |
| **主入口** | 汇总库存明细 **`/inventory/stocks/:stockId`** → 在库明细表 **「手工移库」** → query **`sourceStockItemId`**；建议增加 **`returnStockId`** 便于成功后 `router.replace` 回汇总页并刷新。 |

## 六、库存与事务（语义摘要）

- **单事务**；失败全回滚。  
- **物理路径**：扣减源 `stock_item` / 更新源 `stock`；在目标分桶 **新建** `stock_item`（及必要时新建 `stock`）；**禁止**仅改 `WarehouseId` 破坏分桶与 `StockInItemId` 约束。  
- **凭证路径**：在内核中写入 **虚拟调拨出 + 虚拟调拨入**；数量与 `moveQty` 一致。  
- **并发**：事务内重读或版本校验，避免双开重复迁同一行。

## 七、移库后影响面（回归 checklist）

与历史版本一致，重点包括：

- **`SellOrderItemExtend.PurchasedStock_AvailableQty`**：备货维度变更时调用 `ISellOrderItemPurchasedStockAvailableSyncService`（与现 `ManualStockTransferService` 成功路径一致）。  
- **`pickingtaskitem` 未完成引用**：移库前硬拒绝（与现 Preview 一致）。  
- **`stockledger`**：与数量变更同事务。  
- **前端缓存**：成功回调后 invalidate 或回跳刷新汇总明细接口。

全量检索词：`PurchasedStock`、`pickingtaskitem`、`QtyRepertoryAvailable`、`GetAvailableQtyForSellOrderItem` 等 — 见共享内核与《报关_移库》交叉验证。

## 八、测试与验收（手工）

1. **单扣**：源行可用量减少 = `MoveQty`；目标行增加 = `MoveQty`；**无**重复扣减。  
2. **虚拟凭证**：存在 `stock_out` + `stock_in`，`StockOutType`/`StockInType` 均为 **3**，`SourceId` = `stocktransfer_manual.Id`。  
3. **统计隔离**：销售已出库、采购已入库 **不包含** 上述虚拟单。  
4. **列表与操作**：虚拟出库 **不可** 走「执行出库」真实流程。  
5. **无可迁量 / 拣货占用 / 同仓同库位 / RegionType / 并发双提交** 与历史用例一致。  
6. **回归**：报关模块、执行出库、入库过账 **不受手工移库误伤**。

## 九、实施阶段

| 阶段 | 内容 |
|------|------|
| **P0** | 落地 **`IInternalTransferPostingKernel`**；手工 `ExecuteAsync` **迁入内核**并补齐 **虚拟出库**；全库完成 **类型 3 排除** checklist（共享内核 §五）。 |
| **P1** | 手工移库 **列表/详情 API**、前端列表或筛选、`log_operation` 记录。 |
| **P2** | 形态 B 整仓批量、异步任务、审批（若需要）。 |

**里程碑**（可与内核文档 §九对齐）：内核接口冻结 → 手工切换内核 → 前端列表与筛选 → 报关编排接内核（属报关迭代，见《报关_移库》）。

## 十、风险与待决

1. **`QtyRepertoryAvailable` vs `QtyRepertory`** 作为可迁量定义 — 须与产品、现网占用规则锁定。  
2. **`stock_item` → `stock_in_item` FK**：目标行是否必须虚拟入库 — 以 DB 约束为准（当前手工路径已采用虚拟入库）。  
3. **整仓形态 B** 的性能、失败重试与幂等。  
4. **财务口径**：`StockOutType=3` 在报表中的归属 — 实施前与财务评审签字。

## 十二、从汇总明细到完成移库（端到端任务）

| 步骤 | 说明 |
|------|------|
| S1 | `/inventory/stocks/:stockId` 在库明细点 **手工移库**，带 `sourceStockItemId`（建议同时带 `returnStockId`）。 |
| S2 | `/inventory/transfers-manual` 进入后 **Preview**，失败展示原因与返回汇总按钮。 |
| S3 | 选目标仓/库位、备注 → 二次确认 → `POST` → 成功则 **回跳汇总页并刷新** `getStockItemsForStock`。 |

服务端：`ManualStockTransferService` 组 `InternalTransferPostingRequest` → 内核 `ExecuteAsync` → 返回目标 `stockItemId`、虚拟单号等供前端展示（字段以 DTO 为准）。
