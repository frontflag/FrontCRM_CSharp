# 手工移库功能 — 实施方案（修订：不支持指定数量）

## 〇、前文结论摘要（与报关移库、表职责的关系）

| 结论 | 说明 |
|------|------|
| **与报关移库分离** | 报关移库：前端 **`/inventory/transfers-customers`**，表 **`stocktransfer_customers`** / **`stocktransfer_item_customers`**（**强绑定** `customs_declaration_id`，非空 + 唯一索引）。手工移库：表 **`stocktransfer_manual`** / **`stocktransfer_item_manual`**，前端 **`/inventory/transfers-manual`**，不塞进报关移库页。 |
| **报关「报关完成」与生成移库** | 当前 `CompleteDeclarationAndTransferAsync` **未实现**生成移库与过账；**不能**作为手工移库前置依赖。 |
| **「不支持指定数量」≠ 不改 `stock_item`** | 即使 **整仓迁移** 或 **整行迁移**，仍要对 **`stock`/`stock_item`** 做批量或逐行迁仓/换桶；省掉的是 **「同一行只迁一部分 → 拆行/拆数量」** 的复杂度。 |
| **`stock_in_item`** | **通常不改**：保留历史入库事实；在库现状由 `stock_item` + 移库单体现。若 DB 要求 `stock_item` 必须 FK 到 `stock_in_item`，则采用 **虚拟入库单** 挂接目标新行（与《报关_移库》虚拟入库思路一致）。 |

**本文档需求变更：** 手工移库 **不支持用户填写/选择「指定数量」** — 迁移数量由系统按规则 **整行或整仓一次算死**（见下文两种产品形态）。

## 一、需求边界（修订后）

| 项目 | 一期定义 |
|------|-----------|
| **数量** | **不提供**「移库数量」输入框；**不允许**部分数量移库。系统按所选范围计算应移数量（见 **1.1 两种形态**，二选一或分期做）。 |
| **报关** | 与报关单、报关移库 **无耦合**；不复用报关移库 API/表 `stocktransfer_customers`（除非评审后统一扩表，见第二节备选）。 |
| **目标** | 必选 **目标仓库**；可选 **目标库位**（`LocationId` 语义与现有一致）。 |
| **经济字段** | 不重新计价：目标侧在库行 **继承源行** 采购价、币别、采销关联等（与报关方案「迁库不改价」一致）。 |
| **RegionType** | 目标在库行建议按 **目标仓档案地域** 赋值；未维护则 **沿用源行** 并记录日志（可配置为硬错误）。 |
| **权限与审计** | 独立权限码 + 操作日志（`log_operation` 或等价）；事务执行、失败全回滚。 |

### 1.1 产品形态（需产品拍板，实现二选一或先做 A 后做 B）

**形态 A — 按在库明细「整行迁」（推荐先做）**

- 用户选择 **一条** `stock_item`（检索条件：仓库、物料、库位、`stock_item_code` 等）。
- 系统迁移数量 **固定** = 该行 **`QtyRepertoryAvailable`（或评审定为 `QtyRepertory`）的全部**；若 **占用/销售占用** 导致可迁量 &lt; 在库量，则 **拒绝执行** 或只迁可用量（若选「只迁可用」则仍属系统自动算满可用，**仍无数量输入框**）。
- **不做** 同一物理行上「迁 3 件留 2 件」的拆行。

**形态 B — 整仓批量迁**

- 用户选择 **源仓 + 目标仓**（及可选默认目标库位策略）。
- 系统对该源仓下 **所有满足条件的 `stock_item`** 逐行（或按批）执行与形态 A 相同的 **整行迁** 逻辑。
- **风险**：数据量大、耗时长、需异步任务与进度；一期可不做或仅小仓试点。

**建议：** **一期落地形态 A**；形态 B 列二期或单独「仓间迁移任务」需求。

## 二、数据模型策略（推荐：新建表）

与初版方案一致：**推荐新建** `stocktransfer_manual` / `stocktransfer_item_manual`（库表已建骨架，见迁移 `20260706120000_RenameStockTransferCustomersAndManualTables`），**不**直接复用报关表 `stocktransfer_customers`，避免 `CustomsDeclarationId` 非空、唯一索引及明细上报关/出库通知外键的耦合。

**明细表数量字段：**

- `qty` 可为 **执行时快照**（= 当时整行可迁数量），**非用户录入**；便于审计与对账。

**备选：** 扩展 `stocktransfer_customers` + 可空报关 FK + 部分唯一索引 — **仅**在强需求「所有移库一张表」时评估。

## 三、库存与事务设计（修订：无部分数量）

### 3.1 不变量

- 执行前校验：源行 **可迁量 &gt; 0**；占用规则与现网出库/拣货策略 **对齐**（避免迁走已占用数量）。  
- **单事务**：源 `stock`/`stock_item` 与目标 `stock`/`stock_item` 一致更新 + `stockledger`（类型与财务约定：`STOCK_TRANS` 或 `MANUAL_TRANS`）。  
- 失败 **全回滚**。

### 3.2 源行处理（无「拆出一部分」）

- **不再实现**「qty &lt; 行在库量」的拆行算法。  
- 实现路径仍建议 **扣减源在库 + 目标新建在库行**（与 `StockOutService` 扣减、`InventoryCenterService` 建层逻辑 **公式对齐**），避免 **直接改 `WarehouseId`/`StockAggregateId`** 导致分桶与 `StockInItemId` 唯一性冲突（与《报关_移库》讨论一致）。  
- 迁移数量 = **系统算出的整行可迁量**（整行扣光在库层，目标侧新建一行承接同量）。

### 3.3 目标行与 `stock_in_item`

- 与初版相同：若 `stock_item.stock_in_item_id` **数据库强制非空 FK**，则 **虚拟 `stock_in`/`stock_in_item`** 承载目标新行；**不改历史 `stock_in_item`**。

### 3.4 并发

- 事务内对涉及的 `stock_item` **重读** 或版本校验，防止双开界面重复迁同一行。

## 九、移库后需刷新 / 重算的影响面（实施 checklist）

**原则：** 库存事实以 **`stock` / `stock_item`（及 `stockledger`）** 为准；多数列表接口 **每次读库**，无单独「缓存服务」时 **不必**再发「刷新事件」。以下区分 **「落库正确即自动正确」** 与 **「存在反范式字段须主动重算」**。

### 9.1 落库正确后，读库即一致的模块（回归验证即可）

| 模块 / 能力 | 说明 |
|--------------|------|
| **销售明细「客单在库」可用量** | `InventoryCenterService.GetAvailableQtyForSellOrderItemAsync` 对 `stock` 按 `SellOrderItemId` 汇总 `QtyRepertoryAvailable`；**不依赖**扩展表缓存。移库后只要 `stock`/`stock_item` 数量与绑定关系正确，**下次请求即新值**。API：`GET .../inventory-center/.../available-qty-for-sell-order-line`（见 `InventoryCenterController`）。 |
| **出库申请 / 执行出库** | `StockOutService` 用 `GetAvailableQtyForSellOrderItemAsync` 与拣货绑定 `stock_item_id`；数量来自当前 DB。 |
| **库存中心 / 在库明细 / 汇总库存** | `InventoryCenterService`、库存列表与明细页等以 `stock`/`stock_item` 为源；**无额外聚合表**则无需刷新任务。 |
| **拣货候选 / 按销售行解析库存类型** | `ResolveStockTypesForSellOrderLineAsync`、候选在库行枚举等基于当前 `stock_item`；迁仓后候选仓别会变，**属预期**，无需后台刷新队列。 |

### 9.2 必须在移库事务内或紧后触发的「重算 / 同步」（反范式）

| 数据 | 说明 | 建议动作 |
|------|------|-----------|
| **`SellOrderItemExtend.PurchasedStock_AvailableQty`** | 注释明确：同 **PN+品牌** 下 **备货**（`stock.StockType = 备货`）可用量之和；用于销售列表、出库申请门槛等。**非**每次现算，入库/出库后会调 `ISellOrderItemPurchasedStockAvailableSyncService`。 | 手工移库若改变 **备货** 层 `stock` 的 `QtyRepertoryAvailable`（或跨仓导致 PN+Brand 维度汇总变化），须在过账成功后调用 **`RecalculateByPurchasePnAndBrandAsync(purchasePn, purchaseBrand)`** 或 **`TryRecalculateFromChangedStockInfosAsync(变更的 stock 行)`**（与 `StockOutService` / `StockInService` 一致）。**客单绑定库存（StockType=客单）** 一般 **不**经该字段，但若产品把某类迁仓也纳入备货维度，仍按 PN+Brand 扫一遍更稳。 |
| **`stockledger`** | 若项目要求移库与报关一样留痕，应写入流水（含 from/to 仓、`stock_transfer_id` 或手工单 id 等已有扩列见迁移）。 | 与库存数量同一事务写入，便于财务与对账。 |

### 9.3 业务规则与数据一致性（非「数值重算」但必须检查）

| 项 | 说明 |
|----|------|
| **拣货任务 `pickingtaskitem.stock_item_id`** | 若移库 **源行** 已被拣货任务引用且任务未完成，迁走会导致拣货指向错误仓/无效行。 | 产品规则：**禁止迁** 或 **仅允许迁「无未完成拣货」的在库行**；或迁前自动作废/重绑（复杂，建议一期硬拒绝）。 |
| **出库通知 / 锁** | 若存在与在库行关联的未完成出库流程，同上。 |
| **`StockItem` 派生列** | 如 `SyncDenormalizedComputedFields`（出库状态、入库毛利快照等）在数量变更后是否需调用 — **与出库扣减路径对齐**，避免派生列陈旧。 |
| **前端** | 无服务端缓存时 **用户重新打开页面 / 列表重新请求** 即可；若个别页有 **本地缓存** 或 **SWR 长 stale**，需在操作成功回调里 **invalidate**（实施时扫 `inventoryCenter`、`stock` 相关 composable）。 |

### 9.4 建议实施时的代码检索清单（防漏）

在实现手工移库后，全局检索并做 **集成回归**：

- `PurchasedStock_AvailableQty`、`PurchasedStockAvailableSync`、`GetAvailableQtyForSellOrderItem`
- `QtyRepertoryAvailable`、`StockType`、`StockingSupplement`
- `pickingtaskitem`、`PickingTask`、`stock_item_id`
- `GetFinanceSummaryAsync` / `GetMaterialOverviewAsync`（若移库影响呆滞分析口径）
- 报价/询价页展示「库存」若有独立接口，确认是否读实时库存表

## 十一、移库日志（手工 + 报关 + 后期类型扩展）

### 11.1 目标

- **审计**：谁、何时、哪类移库、源/目标、数量、关联单号（报关单号 / 手工单号）。  
- **统一多种移库**：手工、报关（及未来内部调拨等）**同一套查询模型**或 **可 UNION 的规范字段**。

### 11.2 实现方案（推荐：**专用事件表 + 流水表互补**）

| 层级 | 方案 |
|------|------|
| **库存事实与金额轨迹** | 继续使用 **`stockledger`**（已有 `from_warehouse_id`、`to_warehouse_id`、`stock_transfer_id`、`source_stock_item_id`、`target_stock_item_id` 等扩列，见报关迁移）。移库过账时 **写 ledger**，`BizType` 区分：`STOCK_TRANS`（报关）、**新增** `MANUAL_TRANS` 或 `INTERNAL_TRANS`（与财务/报表评审后固定枚举）。 |
| **业务可读「移库日志」** | 新建 **`inventory_transfer_log`**（命名可评审），建议字段：`id`、`transfer_kind`（enum：`Manual` / `Customs` / `Internal`…）、`business_no`（手工单号或报关移库单号）、`customs_declaration_id`（可空）、`manual_transfer_id`（可空）、`from_warehouse_id`、`to_warehouse_id`、`source_stock_item_id`、`target_stock_item_id`、`qty`、`acting_user_id`、`created_at`、`payload_json`（扩展字段，避免后期改表频繁）。 |
| **与 `log_operation` 关系** | **可选**：重要操作再写一条 `log_operation`（`BizType`+`RecordId`+`OperationDesc`），便于与现有「操作日志」页统一检索；**不必**替代专用表，避免 JSON 塞爆 `OperationDesc`。 |

### 11.3 写入时机

- **手工移库**：事务提交成功后 **同步写** `inventory_transfer_log` + `stockledger`（同一事务优先，保证与库存一致）。  
- **报关移库**：在将来 `CompleteDeclarationAndTransferAsync` 真正落库过账时，同样写 **`transfer_kind=Customs`** + 关联 `stock_transfer_id` / `customs_declaration_id`。  
- **列表 API**：`GET /api/v1/inventory/transfer-logs?kind=&fromDate=&toDate=` 对 `inventory_transfer_log` 分页；报关页若需「只看待报关移库」则 `kind=Customs`。

### 11.4 与现有「操作日志」边界

- **`log_operation`**：偏业务动作（谁改了哪张业务单）。  
- **`inventory_transfer_log`**：偏 **库存位移事实**；与 `stockledger` 一起支撑对账与 WMS 审计。  
- 前端可增加 **「移库日志」** 菜单（权限独立），或并在库存模块下二级入口。

## 十二、`/inventory/transfers-manual` 页面实现方案（由空占位到可执行）

### 12.1 现状与目标

| 项 | 说明 |
|----|------|
| **现状** | `InventoryTransfer.vue` 仅为占位：标题、`sourceStockItemId` query 提示、「敬请期待」文案；**无**表单、**无**调后端接口。 |
| **目标（形态 A）** | 单页完成：**载入源在库行 → 展示只读事实与系统将迁数量 → 选目标仓/库位 → 备注 → 二次确认 → 提交**；全程 **无「移库数量」输入框**（与 §一、§五 一致）。 |

### 12.2 页面布局（建议区块顺序）

1. **页头**：标题（手工移库）、可选「返回」`router.back()` 或链回 `/inventory/stocks/:stockId`（若 query 中带 `returnStockId` 可后续增强）。  
2. **源在库行**  
   - **有** `?sourceStockItemId=`：进入页后自动请求 **预览接口**（见 §12.3），加载失败时 `ElMessage` + 内联错误区，引导用户去 **库存明细列表** 核对或清空 query 重选。  
   - **无** query：提供 **在库明细编号 / 入库单号** 检索入口（调用现有 `GET .../inventory-center/stock-items?...` 分页，选一行后写入本地 `sourceStockItemId` 再触发预览）；或简化为「请从汇总库存明细操作列进入」+ 跳转库存中心（一期可接受）。  
3. **只读摘要**（预览成功后展示）：`stock_item_code`、物料 PN/Brand、源仓/源库位、`qtyRepertory`、`qtyRepertoryAvailable`、**本次将迁数量**（= 服务端按规则算出的整行可迁量，与 §3.2 一致）、若不可迁则展示 **拒绝原因**（占用、未完成拣货、可迁量为 0 等）。  
4. **目标**  
   - **目标仓库**：`el-select`，选项来自现有 `GET .../inventory-center/warehouses`（或项目统一仓库接口）；**过滤规则**：与产品约定（是否允许同仓仅换库位、是否禁止迁到境外等）。  
   - **目标库位**：可选 `el-input` 或字典选择，语义与现网 `LocationId` 一致。  
5. **备注** + **提交**：`el-button` primary；点击后 `ElMessageBox.confirm` 二次确认文案含 **将迁数量、源/目标仓**；确认后 `POST` 执行接口。  
6. **提交成功**：Toast + 可选跳转：返回源 `stock` 汇总明细页或留在本页并清空表单进入下一单。

**UI 技术栈**：与 `StockTransferList.vue`、`InventoryStockDetailPage.vue` 一致（`CrmDataTable` 可选用可不用；表单区用 Element Plus + 现有 `variables.scss` / `btn-primary` 等）。

### 12.3 API 契约（在 §四 基础上落地）

| 方法 | 路径 | 用途 |
|------|------|------|
| `GET` | `/api/v1/inventory/manual-transfers/preview?sourceStockItemId={id}` | **建议新增**：校验源行、计算 **可执行迁出数量**、返回源/仓/占用/拣货拦截原因列表；**不修改**数据库。前端据此决定「提交」是否 `disabled` 及错误展示。 |
| `POST` | `/api/v1/inventory/manual-transfers` | **执行**：Body：`{ "sourceStockItemId", "toWarehouseId", "toLocationId?": null, "remark?": null }`；**无 `qty`**。服务端在事务内过账（`stock`/`stock_item`/`stockledger`/`stocktransfer_manual*` 等，见 §二、§九、§十一），返回新单号、目标 `stockItemId` 等。 |

**取舍说明**：若一期赶工，可 **暂不实现 `preview`**，由 `POST` 在校验失败时返回 **结构化错误**（如 `code: PICKING_BLOCKED`）；但页面体验与「提交前禁用」较弱，**仍推荐单独 Preview**。

**权限**：两接口均要求 `inventory.manual-transfer.execute`（与 §四、§十 一致）；控制器上 `[Authorize]` + 与 RBAC 中间件对齐。

### 12.4 后端实现要点（与现有库存链衔接）

## 十三、从 `/inventory/stocks/:stockId` 点击「手工移库」到完成移库动作（端到端实施方案）

本章在 §三～§四、§十二 基础上，**固定入口**为汇总库存明细页，把 **「点击 → 跳转 → 填单 → 提交 → 落库 → 回跳」** 拆成可开发任务。

### 13.1 入口与路由串联（前端必做）

| 步骤 | 说明 |
|------|------|
| S1 | 用户在 **`/inventory/stocks/:stockId`** 在库明细表点 **「手工移库」**（已实现：带 `sourceStockItemId`）。 |
| S2 | **建议增强 query**（便于回跳与刷新）：除 `sourceStockItemId` 外增加 **`returnStockId`**（= 当前路由的 `stockId`），可选 **`materialId` / `warehouseId` 等** 与现汇总页 query 一致，成功返回后 **`router.replace`** 到同一汇总 URL，用户可立即看到数量变化。 |
| S3 | **`/inventory/transfers-manual`**（`InventoryTransfer.vue`）`onMounted`：若有 `sourceStockItemId` 则 **自动调 Preview**（§12.3）；失败时展示错误，保留「返回汇总明细」按钮（`path: /inventory/stocks/${returnStockId}` + 原 query）。 |

### 13.2 手工移库页内交互（完成「具体动作」前的 UI）

与 §12.2 一致，补充 **与汇总页的衔接**：

1. **只读区**：展示预览返回的源仓、在库行编号、**将迁数量**（只读）、拦截原因（若有）。  
2. **目标仓 / 目标库位 / 备注**：用户填写；**提交**前 `ElMessageBox.confirm`，文案必须含 **将迁数量** 与 **源→目标仓**。  
3. **提交**：`POST /api/v1/inventory/manual-transfers`；`submitting` 期间禁用按钮防双点。  
4. **成功**：`ElMessage.success` + **优先** `router.push` 回 **`/inventory/stocks/{returnStockId}`**（并带上原 summary query），汇总页 `onActivated`/`watch stockId` 触发 **重新拉取** `getStockItemsForStock`，用户即看到更新后的在库明细。  
5. **失败**：展示后端 `message` / 结构化 `code`（如拣货占用）；不清空 `sourceStockItemId`，便于用户改目标仓重试。

### 13.3 服务端「移库动作」执行顺序（`ExecuteAsync` 建议伪代码）

**单数据库事务**（失败全回滚），与 §3.2「扣源 + 目标新建」一致：

1. **加锁 / 重读**：按 `sourceStockItemId` 加载 `StockItem`（及关联 `Stock`/`StockInfo` 若需要），校验行仍存在且未删除。  
2. **业务拒绝**（任一命中则抛业务异常）：  
   - 未完成拣货引用（§9.3）；  
   - 可迁量 ≤ 0（定义与 §八 一致，如 `QtyRepertoryAvailable`）；  
   - 目标仓非法、与产品规则冲突（如禁止跨地域，若已定义）。  
3. **计算 `moveQty`**：整行可迁量，**非用户传入**；写入 `stocktransfer_item_manual.Qty` 快照。  
4. **生成主单**：`stocktransfer_manual`（`TransferCode` 流水号、`FromWarehouseId`/`ToWarehouseId`、`Status`、`Remark`、`Confirmed*` 等按状态机约定）。  
5. **库存过账**（与现有入库/出库公式对齐，避免直接改 `WarehouseId` 破坏分桶，见 §3.2）：  
   - 扣减源 `stock_item`（及同步更新对应 `stock`/`stock_info` 数量与可用量）；  
   - 在 **目标分桶** 下 **新建** `stock_item`（及必要时新建/合并 `stock` 层）；**经济字段继承源行**（§一）；**`RegionType`** 按目标仓档案或 §一约定赋值。  
   - 若 `stock_item.stock_in_item_id` 非空且目标行必须挂 FK：走 **虚拟 `stock_in`/`stock_in_item`**（§一、§八）。  
6. **流水**：`stockledger` + `BizType`（如 `MANUAL_TRANS`，与财务评审一致）；关联 `stocktransfer_manual` 主键写入扩列（若已有 `stock_transfer_id` 语义可区分报关/手工，见 §十一）。  
7. **反范式**：若涉及备货层 PN+Brand 可用量变化，事务提交后（或同一事务末，按现有服务是否要求已提交读）调用 **`ISellOrderItemPurchasedStockAvailableSyncService`**（§9.2）。  
8. **`StockItem` 派生列**：若项目存在 `SyncDenormalizedComputedFields` 等，与出库扣减路径 **同一调用点** 更新（§9.3）。  
9. **提交**：`SaveChanges` 一次；返回 **手工单号、目标 `stockItemId`、实际 `moveQty`** 供前端展示与回跳。

### 13.4 权限与审计

- API：`inventory.manual-transfer.execute`（§四）。  
- 可选：`log_operation` + 专用移库日志表（§十一）；与 `POST` 同事务或紧后同步写。

### 13.5 实施任务拆分（便于排期）

| 任务 ID | 内容 | 依赖 |
|---------|------|------|
| T1 | `InventoryStockDetailPage`：`goManualTransfer` 增加 `returnStockId`（等 query） | 无 |
| T2 | `ManualStockTransfersController` + `preview` DTO | 无 |
| T3 | `IManualStockTransferService.ExecuteAsync` 过账核心 + 单元/集成测试 | T2、表已存在 |
| T4 | `InventoryTransfer.vue` 表单 + `manualStockTransfer.ts` + 成功回跳刷新 | T2、T3 |
| T5 | RBAC 权限节点注册 + 前端按钮 `v-permission`（若项目统一） | T4 |

### 13.6 验收（针对本入口）

- 从 **`/inventory/stocks/:id`** 点手工移库 → 完成一次迁出后，回到 **同一 `id`** 页，在库明细 **数量与仓别** 与预期一致；**无**未完成拣货时可迁，**有**则拒绝且提示明确。  
- 备货行迁仓后，销售侧 **备货可用** 与 §9.2 一致（抽样接口或 DB 字段）。
