# 销售订单明细编号方案

## 1. 概述

### 1.1 设计目标

- 每条 **销售订单明细**（`sellorderitem`）拥有独立的业务编号：**销售订单明细编号**。
- 编号规则：**`销售单号 + "-" + Item 序号`**（与 `SellOrderCode` 一致的前缀）。
- **Item 序号**在同一销售订单内：**从 1 起、递增、可跳号、不重复**；**已删除或替换掉的明细曾用过的序号不得复用**。
- 通过新建 **`sellorderextend` 表** 的 **`last_item_line_seq`** 字段维护流水水位，并配合数据库行锁，解决 **不重复** 与 **并发竞争** 问题。
- 明细表增加 **`sell_order_item_code`**，持久化完整编号字符串，供列表、详情及外部引用。

### 1.2 表职责划分

| 表 | 职责 |
|----|------|
| **`sellorder`** | 销售订单主数据：单号、客户、业务员、状态、金额汇总、交货信息等，**仅与订单本身直接相关**的字段。 |
| **`sellorderextend`**（新建） | **1:1** 对应一条 `sellorder`，承载**订单销售信息之外的扩展信息**；本方案中至少包含 **`last_item_line_seq`**（明细序号流水号）。后续可扩展：外部单号、审批扩展、统计缓存等。 |
| **`sellorderitem`** | 订单明细；新增 **`sell_order_item_code`** 存储完整「销售订单明细编号」。 |

---

## 2. 数据模型要点

### 2.1 `sellorderextend`（新建）

- **主键**：`SellOrderId`（`varchar(36)`），与 `sellorder.SellOrderId` **一致**，并建立 **外键**，保证 **一条主单对应一条扩展记录**。
- **必选字段（本方案）**  
  - **`last_item_line_seq`**：`integer NOT NULL DEFAULT 0`  
    - 语义：本单**已发放过的明细 Item 序号最大值**。  
    - **只增不减**：删除明细、整单替换明细时**不得**将该值减小。  
    - 用途：新行分配序号时 **`nextSeq = last_item_line_seq + 1`**（或批量 **`+k`** 后一次写回）。

可选：与项目其它扩展表对齐增加 `CreateTime` / `ModifyTime` 等审计字段。

### 2.2 `sellorderitem`（变更）

- **`sell_order_item_code`**：`varchar(48～64)`（建议略大于 `SellOrderCode` 最大长度 + 分隔符 + 序号位数）。  
- 存储格式：**`{SellOrderCode}-{Item序号}`**，例如 `SO2604010001-7`。  
- 建议约束：**`UNIQUE (sell_order_id, sell_order_item_code)`**，防止重复编号。

---

## 3. 销售订单明细编号规则（合并说明）

### 3.1 组成

- **销售订单明细编号** = **`SellOrderCode` + `"-"` + `Item 序号`**。  
- **`Item 序号`**：正整数；**每单独立编号**；从 1 开始；**顺序递增**；**允许跳号**（历史占用后删除即形成跳号）；**同一单内不可重复**；**已删除明细曾用序号永不复用**（由 `last_item_line_seq` 保证）。

### 3.2 双字段分工

| 存储位置 | 字段 | 作用 |
|----------|------|------|
| `sellorderextend` | `last_item_line_seq` | **权威流水水位**：分配新序号的依据；配合行锁处理并发。 |
| `sellorderitem` | `sell_order_item_code` | **对外展示与业务引用**的完整编号。 |

**线上分配不以「扫描全表明文解析 max」为唯一依据**；解析仅建议用于 **迁移回填、数据校验与运维修复** 时的辅助对账。

---

## 4. 分配算法（新建明细）

在**同一数据库事务**内执行：

1. 按 **`SellOrderId`** 定位 **`sellorderextend`**：  
   - 若不存在（如新单首次写扩展）：**插入**一行，`last_item_line_seq = 0`（或与迁移策略一致）。  
2. **锁定扩展行**（避免并发拿到同一序号），例如：  
   - `SELECT … FROM sellorderextend WHERE "SellOrderId" = @id FOR UPDATE`（或与 ORM/工作单元等价的行锁）。  
3. 计算序号：  
   - 单条：`nextSeq = last_item_line_seq + 1`；  
   - 同一请求插入 **`k`** 条：使用 **`nextSeq, nextSeq+1, …, nextSeq+k-1`**，然后 **`last_item_line_seq += k`**。  
4. 读取 **`sellorder.SellOrderCode`**（同事务内应已可见）。  
5. 写入每条 **`sellorderitem`**：**`sell_order_item_code = SellOrderCode + "-" + 序号`**。  
6. **`UPDATE sellorderextend`** 写回 **`last_item_line_seq`**。

**说明**：

- **物理删除明细**或**整单替换明细（先删后插）**：只要**不降低** `last_item_line_seq`，新行序号从水位继续递增，**不会复用**历史序号。  
- **并发**：第二笔请求在扩展行锁上等待，顺序获得递增序号，**不会撞号**。

---

## 5. 生命周期与一致性

### 5.1 新建销售订单

1. 插入 **`sellorder`**。  
2. **插入 `sellorderextend`**（`last_item_line_seq = 0`）。  
3. 按条插入 **`sellorderitem`**，每条按第 4 节递增水位并写入 **`sell_order_item_code`**。

### 5.2 编辑销售订单（替换明细）

- 删除旧 **`sellorderitem`** 时：**禁止**将 **`sellorderextend.last_item_line_seq`** 减小。  
- 新插入的明细仍按 **当前水位 +1、+2…** 分配序号并写入 **`sell_order_item_code`**。

### 5.3 删除整张销售订单

- 若业务 **物理删除主单** 且级联删明细：应 **同时删除 `sellorderextend`**，保持 1:1 生命周期一致。  
- 若为 **作废/关单** 不删行：主单与扩展表一并保留，水位保留，无额外问题。

---

## 6. 索引与约束建议

- **`sellorderextend`**：`PRIMARY KEY ("SellOrderId")`，`FOREIGN KEY` → `sellorder`。  
- **`sellorderitem`**：**`UNIQUE (sell_order_id, sell_order_item_code)`**。  
- 若需按编号检索，可再评估 **`sell_order_item_code`** 单列索引（按实际查询决定）。

---

## 7. 历史数据迁移（概要）

1. 创建表 **`sellorderextend`**，为**每条现有 `sellorder`** 插入一行，`last_item_line_seq = 0`。  
2. **`sellorderitem` 增加 `sell_order_item_code`**。  
3. 按 **`sell_order_id`** 分组，对现有明细按 **`CreateTime`、主键** 排序，生成序号 **`1..n`**，写入 **`{SellOrderCode}-{i}`**。  
4. **`UPDATE sellorderextend`**：每单 **`last_item_line_seq = 该单当前最大 Item 序号`**。  
5. 将 **`sell_order_item_code`** 设为 **NOT NULL**（迁移完成后），并创建 **唯一约束**。  
6. 建议同时提供 **`scripts/ensure_*.sql`** 与 **EF Migration**（与仓库惯例一致）。

**说明**：若历史上曾 **物理删除** 过明细，迁移无法还原已占用序号；迁移后 **`last_item_line_seq`** 应取 **`max(按现存行回填的最大序号, 如有对账需求再与业务确认)`**，保证后续新号从正确水位继续。

---

## 8. 解析规则的定位（可选）

若需对 **`sell_order_item_code`** 做解析以提取 **Item 序号**（例如运维脚本）：

- 约定格式：**`{SellOrderCode}-{正整数}`**，且 **序号位于最后一个 `-` 之后** 的整段数字。  
- **`SellOrderCode` 可含多个 `-`** 时，仍以 **最后一段为序号** 为准。  
- 解析失败的数据应单独标记修复，**不参与**线上分配逻辑。

---

## 9. API 与前端（实现阶段）

- 明细 DTO 增加 **`sellOrderItemCode`**（映射 `sell_order_item_code`）。  
- **销售订单明细列表**、**销售订单详情** 等：第一列或显著位置展示「销售订单明细编号」。  
- **`sellorderextend`** 若暂无其它对外字段，可先 **不暴露** 独立接口，仅服务层内部维护；待扩展字段成熟后再在详情/编辑中展示。

---

## 10. 测试要点（实现阶段）

- 新单首条明细 → `…-1`；同一请求批量插入 → `…-1` … `…-N`。  
- 已有明细后再加行 → **`last_item_line_seq + 1`**。  
- **替换明细**后新行序号 **大于** 替换前曾出现的最大序号（构造删行场景验证）。  
- **并发**两请求对同一单各加一行 → 编号不重复、无死锁（或合理超时重试）。  
- **`SellOrderCode` 含 `-`** 时，存储与展示仍符合约定。

---

## 11. 方案小结

1. **`sellorderextend`**：与 **`sellorder` 1:1**，承载订单外扩展信息；**`last_item_line_seq`** 为本单 **明细 Item 序号流水水位**，配合 **行锁**，保证 **不重复与并发安全**。  
2. **`sellorderitem.sell_order_item_code`**：保存完整 **`{SellOrderCode}-{序号}`**；序号 **仅由扩展表水位递增产生**，不依赖删除后重新扫描明细解析 max 作为权威来源。
