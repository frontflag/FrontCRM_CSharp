# 移库共享内核（虚拟出 + 虚拟入）— 实施方案

> **目标**：报关移库与手工移库均落 **虚拟出库单 + 虚拟入库单**（与真实购销出入库区分），并将 **「虚拟出 + 虚拟入 + 扣源 + 加目标 + ledger + 防双扣」** 收敛为 **无业务语义 / 仅枚举上下文** 的 **共享内核**；报关 / 手工两层只做 **参数组装、前置校验、写各自主表**。  
> **硬约束**：**严禁双扣**；**严禁**在列表、报表、扩展字段重算、拣货/收款链路中把虚拟单当作真实购销出入库。

---

## 一、设计原则（必须写进评审与 Code Review）

### 1.1 单一事实来源（防双扣）

| 规则 | 说明 |
|------|------|
| **库存数量唯一写口** | 对 `stock` / `stock_item` 的 **`Qty*` 变更**（含 `QtyStockOut`、`QtyRepertory`、`QtyRepertoryAvailable` 等）**只允许**发生在共享内核 `InternalTransferPostingKernel`（名称可评审）内，且 **每条移库事务内对每个 `stock_item_id` 至多执行一次扣减、一次增加**。 |
| **禁止** | 虚拟出库 / 虚拟入库 **不得** 再调用 `StockOutService.Execute…`、`StockInService` 正常过账入口去二次改库存（否则必双扣）。 |
| **虚拟单角色** | `stock_out` / `stock_in` 及其明细在本方案中为 **凭证与审计镜像**：数量与内核已执行的 `moveQty` **一致**，用于单证对称、列表筛选、关联 `SourceId`/`SourceCode` 回溯移库主单。 |

### 1.2 与真实购销出入库隔离（防误用）

实体注释已约定（`CRM.Core/Models/Inventory/StockInfo.cs`）：

- **`StockInType`**：`1` 采购、`2` 退货、`**3` 调拨入库**`、`4` 其他。  
- **`StockOutType`**：`1` 销售、`2` 退货、`**3` 调拨出库**`、`4` 其他。

**统一约定**：

- 移库虚拟入：`StockInType = **3**`（调拨入库），`Remark` 带固定前缀如 `【移库虚拟入库】` + 移库业务单号。  
- 移库虚拟出：`StockOutType = **3**`（调拨出库），`Remark` 带固定前缀如 `【移库虚拟出库】` + 移库业务单号。  
- **头表关联**：`StockOut.SourceId` / `StockIn.SourceId`（及可选 `SourceCode`）**必须**指向 **移库主单主键**（`stocktransfer_manual.Id` 或 `stocktransfer_customers.Id`），便于追溯与对账。

**全库检索清单（实施时逐项改）**：

- 凡统计 **「销售出库」**、**「已实际出库数量」**（如 `SellOrderItemExtend` 注释口径）、**「采购入库累计」**（`PurchaseOrderItemExtend` 等），**必须**显式限定 `StockOutType = 1` / `StockInType = 1`（或等价白名单），**不得**用「非草稿即算」这类模糊条件。  
- **`StockOutService` / `StockInService` 列表、执行出库、质检完成入库** 等入口：对 `StockOutType=3` / `StockInType=3` 且 `SourceId` 指向移库主单的记录 **拒绝执行/删除/改状态**（或统一隐藏操作按钮），避免走真实业务状态机。  
- **前端**：出库单列表、入库单列表默认 **排除调拨虚拟**（或单独 Tab「调拨/移库凭证」）；若合并展示，须强标签区分。

## 二、分层架构

```
┌─────────────────────────────────────┐
│ ManualStockTransferService          │  前置：拣货占用、Preview、写 stocktransfer_manual*
│ CustomsTransferOrchestrator         │  前置：报关状态、结关、1:1:1、写 stocktransfer_customers*
└──────────────┬──────────────────────┘
               │ 组装 InternalTransferPostingRequest（无业务名，仅数据）
               ▼
┌─────────────────────────────────────┐
│ IInternalTransferPostingKernel      │
│   ExecuteAsync(request, ct)         │  唯一库存写口 + 虚拟出/入 + ledger
└─────────────────────────────────────┘
```

- **内核**：新项目 `CRM.Core/Services/InternalTransfer/`（路径可评审）下 `InternalTransferPostingKernel` + `InternalTransferPostingRequest` / `Result`。  
- **编排**：现有 `ManualStockTransferService` 精简为编排；报关在 `CompleteDeclarationAndTransferAsync` 实现体中调用同一内核（从 `CustomsDeclarationService` 或专用服务调用均可，**禁止**在 Controller 写库存 SQL）。

## 三、`InternalTransferPostingRequest`（建议字段，无业务语义）

| 字段 | 说明 |
|------|------|
| `Kind` | 枚举：`Manual` \| `Customs`（仅用于 **Remark 模板、Ledger BizType、操作日志 BizType**；不参与数量公式）。 |
| `MoveQty` | 已在外层算好的 **正整数**；内核内 **禁止再读「用户输入 qty」**（与现手工移库一致）。 |
| `SourceStockItemId` | 源在库行。 |
| `FromWarehouseId` / `ToWarehouseId` | 冗余校验：须与源行、目标仓一致。 |
| `ToLocationId` | 可空；与现手工规则一致。 |
| `TransferHeaderId` | 移库主单主键（手工或报关表），写入虚拟出/入 `SourceId`、ledger `BizId`。 |
| `TransferBusinessCode` | 展示用单号（`TransferCode` / 报关移库单号），写入 `SourceCode`、Remark。 |
| `LedgerBizType` | 如现有 `StockLedgerBizType.ManualTransfer` / `StockLedgerBizType` 报关枚举（与财务评审一致）。 |
| `ActingUserId` | 操作人。 |
| `LineSnapshots` | **报关多行时**为 `List<LineSnapshot>`（每行：源 `stock_item_id`、`moveQty`、金额快照）；手工可为单元素列表。内核支持 **同一事务多行顺序执行**（每行仍满足单写口）。 |

**内核输出**：

- `VirtualStockOutId`、`VirtualStockInId`、目标 `TargetStockItemId`、目标 `TargetStockAggregateId`、各 `StockOutCode`/`StockInCode`（便于前端展示）。  
- 供编排层写 **`stocktransfer_item_customers`** 等明细行引用。

## 四、内核 `ExecuteAsync` 事务步骤（顺序固定，防双扣）

> 以下顺序 **不得** 随意调整；任何一步失败 **整单回滚**。

1. **加锁与重读**：按 `sourceStockItemId`（或多行逐条）加载 `StockItem` + `StockInfo`，校验未软删、可用量仍 ≥ `MoveQty`。  
2. **二次校验**（编排未做或需内核兜底）：未完成拣货引用等（与现 `ManualStockTransferService` 一致）。  
3. **解析/创建目标分桶 `StockInfo`**（与现手工逻辑一致：`StockMatchesInboundBucket` 等）。  
4. **物理扣源**：与现 `ApplyManualTransferOut` 等价逻辑 **仅执行一次**（更新源 `stock_item` / `stock`）。  
5. **物理加目标**：新建或更新目标 `stock_item`（与现手工一致）；若需 FK，目标行仍挂 **虚拟入库明细**。  
6. **写虚拟出库**（凭证）：`stock_out` + `stock_out_item`  
   - `StockOutType = 3`，`WarehouseId = FromWarehouseId`，`TotalQuantity = MoveQty`，`Status = 2`（已出库，与「凭证已生效」语义一致，**不走**待拣货流程）。  
   - `SourceId = TransferHeaderId`，`SourceCode = TransferBusinessCode`。  
   - 明细 `stock_out_item` 与源物料、数量、`stock_item` 引用（可填 `SellOrderItemId` 等冗余 **仅展示**，**不得**触发销售出库汇总逻辑——见 §五）。  
7. **写虚拟入库**（凭证）：`stock_in` + `stock_in_item`  
   - `StockInType = 3`，`WarehouseId = ToWarehouseId`，`Status = 2`（或项目对「已入库」约定值，与现手工一致）。  
   - `SourceId` / `SourceCode` 同上。  
   - 目标新 `stock_item` 的 `StockInItemId` 指向本条虚拟明细。  
8. **写 `stockledger`**：一条或多条（与现 `ManualStockTransferService` 对齐；报关多行可多行 ledger），`BizType` 来自 `Request.LedgerBizType`。  
9. **反范式与扩展**：`TryRecalculateFromChangedStockInfosAsync`、`SyncDenormalizedComputedFields` 等与 **现手工移库成功路径** 完全一致。  
10. **`SaveChanges` 单次提交**（或 UnitOfWork 一次提交）。

**关键**：步骤 4～7 中 **步骤 4、5 是唯一改变 `stock`/`stock_item` 数量的步骤**；步骤 6、7 **不得** 调用会改库存的公共服务。

## 五、与扩展字段、统计的硬隔离（防误用 — 实施 checklist）

| 位置 | 动作 |
|------|------|
| `SellOrderItemExtend` / 销售「已出库」累计 | 仅 `StockOutType == 1`；**排除** `3`。 |
| `PurchaseOrderItemExtend` / 采购「已入库」累计 | 仅 `StockInType == 1`；**排除** `3`。 |
| `StockOutService.GetStockOutListAsync` 等 | 默认 `WHERE stock_out_type <> 3 OR (type=3 AND source_id IS NULL)` — **以评审为准**；推荐 **默认列表排除 3**，另提供「调拨凭证」筛选。 |
| 入库列表 API | 同上。 |
| 拣货 / 出库执行 | 校验：若 `stock_out.StockOutType=3` 且 `SourceId` 为移库主单 → **禁止**绑定拣货任务。 |
| 财务 / 收入报表 | 明确排除 `StockOutType=3`（除非单独做「内部移库成本」专题）。 |

## 六、手工移库编排改造要点

1. 保留 `PreviewAsync`、拣货校验、`stocktransfer_manual` / `item` 写入。  
2. `ExecuteAsync` 改为：构建 `InternalTransferPostingRequest { Kind=Manual, LedgerBizType=ManualTransfer, … }` → 调内核 → 将内核返回的虚拟出/入主键 **可选** 回写到扩展表（若需要：可增加 `stocktransfer_item_manual` 上 `VirtualStockOutId`/`VirtualStockInId` 列，**非必须**，`SourceId` 已可反查）。  
3. **删除**当前「仅虚拟入、无虚拟出」的重复逻辑，避免与内核分叉。

---

## 七、报关移库编排改造要点

1. 实现 `CompleteDeclarationAndTransferAsync` 中 **占位抛错** 之后的真实逻辑：校验报关状态、无重复移库单、按报关明细 **逐行或批量** 构建 `InternalTransferPostingRequest.LineSnapshots`。  
2. 先写 **`stocktransfer_customers` 头**（或先写头再循环内核 — 需统一 `TransferHeaderId`），再对每一行调用内核 **或** 内核一次接收多行（推荐 **一次事务多行** 减少中间状态）。  
3. `LedgerBizType` 使用报关约定（如 `STOCK_TRANS`），`Kind=Customs`，Remark 模板区分。  
4. 与 **出库通知 / 报关单 1:1:1** 的约束在编排层完成，内核不解析报关业务。

---

## 八、测试用例（验收必过）

1. **单扣验证**：移库前后源 `stock_item` 可用量差 = `MoveQty`；数据库中 **不存在** 两条不同业务单对同一 `stock_item_id` 重复扣减。  
2. **虚拟单不进销售出库统计**：构造一单虚拟 `StockOutType=3`，跑销售已出库汇总 → **不得**增加。  
3. **虚拟单不进采购入库累计**：同上 `StockInType=3`。  
4. **列表默认不可当普通出库执行**：虚拟出库在 UI 无「执行出库」或 API 返回 400。  
5. **报关 + 手工** 各跑一条回归：ledger、`stocktransfer_*`、目标行 `stock_in_item_id` 正确。  
6. **并发**：双请求同一 `sourceStockItemId` → 其一成功其一失败，无负库存。
