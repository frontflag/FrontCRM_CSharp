# 采购订单明细编号方案

> 本方案与《[销售订单明细编号](./销售订单明细编号.md)》**同理**：主单扩展表 1:1 维护 **`last_item_line_seq`**，明细表存完整 **`{采购单号}-{Item序号}`**，解决不重复、跳号与并发。

---

## 1. 概述

### 1.1 设计目标

- 每条 **采购订单明细**（`purchaseorderitem`）拥有独立的业务编号：**采购订单明细编号**。
- 编号规则：**`采购单号 + "-" + Item 序号`**（前缀与 `PurchaseOrderCode` 一致）。
- **Item 序号**在同一采购订单内：**从 1 起、递增、可跳号、不重复**；**已删除或替换掉的明细曾用过的序号不得复用**。
- 通过新建 **`purchaseorderextend` 表** 的 **`last_item_line_seq`** 维护流水水位，并配合数据库 **行锁**，解决 **不重复** 与 **并发竞争**。
- 明细表增加 **`purchase_order_item_code`**（或项目统一命名下的等价列），持久化完整编号，供列表、详情及外部引用。

### 1.2 与现有「明细扩展」的关系

- 项目中已有 **`purchaseorderitemextend`**：**一行采购明细一行扩展**（与 `PurchaseOrderItem` 1:1），用于入库/付款/进度等**行级**字段。  
- **`purchaseorderextend`**：**一行采购主单一行扩展**，用于**主单级**附加信息与**本单明细序号水位**；二者层级不同，**并存、不替代**。

### 1.3 表职责划分

| 表 | 职责 |
|----|------|
| **`purchaseorder`** | 采购订单主数据：单号、供应商、采购员、状态、金额汇总等，**仅与订单本身直接相关**的字段。 |
| **`purchaseorderextend`**（新建） | **1:1** 对应一条 `purchaseorder`，承载**主单业务之外的扩展信息**；本方案至少含 **`last_item_line_seq`**（明细 Item 序号流水号）。后续可扩展：外部 PO 号、对账标记等。 |
| **`purchaseorderitem`** | 采购明细；新增 **`purchase_order_item_code`** 存储完整「采购订单明细编号」。 |
| **`purchaseorderitemextend`**（已有） | 明细行级扩展，与本方案的「明细编号」列**正交**。 |

---

## 2. 数据模型要点

### 2.1 `purchaseorderextend`（新建）

- **主键**：`PurchaseOrderId`（`varchar(36)`），与 `purchaseorder.PurchaseOrderId` **一致**，**外键** 指向主单，保证 **一条主单对应一条扩展记录**。  
- **必选字段（本方案）**  
  - **`last_item_line_seq`**：`integer NOT NULL DEFAULT 0`  
    - 语义：本单**已发放过的明细 Item 序号最大值**。  
    - **只增不减**：删除明细、整单替换明细时**不得**减小。  
    - 用途：新行 **`nextSeq = last_item_line_seq + 1`**（或批量 **`+k`** 后一次写回）。

可选：与 `sellorderitemextend` / 其它扩展表对齐增加 `CreateTime` / `ModifyTime` 等审计字段。

### 2.2 `purchaseorderitem`（变更）

- **`purchase_order_item_code`**：`varchar(48～64)`（建议略大于 `PurchaseOrderCode` 最大长度 + 分隔符 + 序号位数；当前主单号多为 `varchar(32)`）。  
- 存储格式：**`{PurchaseOrderCode}-{Item序号}`**，例如 `PO2604010001-7`。  
- 建议约束：**`UNIQUE (purchase_order_id, purchase_order_item_code)`**（列名以实际数据库为准），防止重复编号。

---

## 3. 采购订单明细编号规则

### 3.1 组成

- **采购订单明细编号** = **`PurchaseOrderCode` + `"-"` + `Item 序号`**。  
- **`Item 序号`**：正整数；**每单独立**；从 1 开始；**递增**；**可跳号**；**同一单内不重复**；**已删除明细曾用序号永不复用**（由 `purchaseorderextend.last_item_line_seq` 保证）。

### 3.2 双字段分工

| 存储位置 | 字段 | 作用 |
|----------|------|------|
| `purchaseorderextend` | `last_item_line_seq` | **权威流水水位**；配合行锁处理并发。 |
| `purchaseorderitem` | `purchase_order_item_code` | **对外展示与业务引用**的完整编号。 |

**线上分配不以「扫描全表明文解析 max」为唯一依据**；解析仅用于 **迁移回填、校验与运维修复**。

---

## 4. 分配算法（新建明细）

在**同一数据库事务**内：

1. 按 **`PurchaseOrderId`** 定位 **`purchaseorderextend`**；不存在则 **INSERT**，`last_item_line_seq = 0`。  
2. **锁定扩展行**（如 `SELECT … FOR UPDATE`），避免并发同号。  
3. **`nextSeq = last_item_line_seq + 1`**；同一请求插入 **`k`** 条则用连续 **`k`** 个序号，并 **`last_item_line_seq += k`**。  
4. 读取 **`purchaseorder.PurchaseOrderCode`**。  
5. 写入每条 **`purchaseorderitem`**：**`purchase_order_item_code = PurchaseOrderCode + "-" + 序号`**。  
6. **`UPDATE purchaseorderextend`** 写回 **`last_item_line_seq`**。

**物理删除明细**或**先删后插的整单替换**：只要不降低水位，新序号**不会复用**历史序号。**并发**下第二请求在锁上排队，顺序取号。

---

## 5. 生命周期与一致性

### 5.1 新建采购订单

1. 插入 **`purchaseorder`**。  
2. 插入 **`purchaseorderextend`**（`last_item_line_seq = 0`）。  
3. 插入 **`purchaseorderitem`**（及既有 **`purchaseorderitemextend`** 逻辑），每条按第 4 节分配编号。

### 5.2 编辑采购订单（替换明细）

- 删除旧明细时：**禁止**减小 **`purchaseorderextend.last_item_line_seq`**。  
- 新明细从当前水位继续递增。

### 5.3 删除整张采购订单

- 物理删除主单且级联删明细时：**同时删除 `purchaseorderextend`**，保持 1:1 生命周期一致。

---

## 6. 索引与约束建议

- **`purchaseorderextend`**：`PRIMARY KEY ("PurchaseOrderId")`，`FOREIGN KEY` → `purchaseorder`。  
- **`purchaseorderitem`**：**`UNIQUE (purchase_order_id, purchase_order_item_code)`**。

---

## 7. 历史数据迁移（概要）

1. 建 **`purchaseorderextend`**，为每条 **`purchaseorder`** 插入一行，`last_item_line_seq = 0`。  
2. **`purchaseorderitem` 增加 `purchase_order_item_code`**。  
3. 按 **`purchase_order_id`** 分组，明细按 **`CreateTime`、主键** 排序，赋 **`1..n`**，写入 **`{PurchaseOrderCode}-{i}`**。  
4. **`UPDATE purchaseorderextend`**：每单 **`last_item_line_seq = 该单最大 Item 序号`**。  
5. **NOT NULL**、**唯一约束**；提供 **`scripts/ensure_*.sql`** 与 **EF Migration**。

**说明**：历史曾物理删除过的明细无法还原已占序号；迁移后水位应 **≥ 现存行最大序号**。

---

## 8. 解析规则（可选、运维/对账）

- 约定：**`{PurchaseOrderCode}-{正整数}`**，**序号**为 **最后一个 `-` 之后**的整段数字（`PurchaseOrderCode` 可含多个 `-`）。  
- 解析失败的数据单独修复，**不参与**线上分配。

---

## 9. API 与前端（实现阶段）

- 明细 DTO 增加 **`purchaseOrderItemCode`**（映射 `purchase_order_item_code`）。  
- **采购订单明细列表**、**采购订单详情** 等展示「采购订单明细编号」。  
- **`purchaseorderextend`** 若无其它对外字段，可先仅服务层维护。

---

## 10. 测试要点（实现阶段）

- 新单首条 → `…-1`；批量插入 → 连续序号。  
- 有历史行后再加行 → 水位 +1。  
- 替换明细后不复用已占用序号。  
- 并发同单加行不重复、无错误覆盖水位。  
- **`PurchaseOrderCode` 含 `-`** 时格式仍正确。

---

## 11. 方案小结

1. **`purchaseorderextend`**：与 **`purchaseorder` 1:1**；**`last_item_line_seq`** 为明细 Item 序号**权威水位**，**行锁**保证并发安全。  
2. **`purchaseorderitem.purchase_order_item_code`**：完整 **`{PurchaseOrderCode}-{序号}`**；序号**仅由水位递增产生**。  
3. 与 **`purchaseorderitemextend`（行级扩展）** 分工清晰：**主单扩展管水位与主单杂项，行扩展管行级业务字段**。
