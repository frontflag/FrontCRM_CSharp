# 库存 · 拣货 · 出库 整改方案

> 本文档汇总业务逻辑与实施方案，与对话中已确认口径一致；**历史无 `stock_item_id` 的拣货数据不兼容新执行出库**，旧执行路径废弃；异常可由**管理员**人工处理。

---

## 一、背景与问题

### 1.1 现状问题

1. **执行出库与拣货脱节**  
   `ExecuteStockOutAsync` 在校验「已生成并完成拣货任务」后，仍按 `request.Items` 与全量库存 **重新 FIFO** 选桶/选明细，**不读取** `pickingtaskitem`，与拣货分配可能不一致。

2. **溯源不足**  
   `stockout` 无法明确对应**哪一次拣货任务**；`PickerId` / `PickedTime` 在业务代码中基本未赋值，且语义为「拣货人 / 拣货完成时间」，**不宜**挪作存拣货任务 Id。

3. **明细粒度不足**  
   若一次拣货涉及多条在库明细，需在 **`pickingtaskitem` 1:N** 上表达，每条绑定一条 **`stockitem`**。

4. **FIFO 定位**  
   FIFO 仅作为**待拣货列表的排序建议**；**具体拣哪几条 `stockitem`、各多少数量，由物流人工确认**。

### 1.2 目标

- **执行出库严格以拣货结果为准**：按已完成拣货任务的 `pickingtaskitem`（含 `stock_item_id`、数量）扣账并生成 `stockoutitem`。  
- **出库单可追溯拣货动作**：`stockout` 记录 `picking_task_id`；`stockoutitem` 记录 `picking_task_item_id`（推荐）。  
- **完成拣货后明细冻结**：未拣满禁止点「完成拣货」；完成后**不允许**再改拣货行。  
- **兼容策略**：仅支持新流程；历史已完成拣货但无 `stock_item_id` 的，**不允许**再走新执行出库；旧自动扣账路径**废弃**。

## 二、业务逻辑

### 2.1 角色与对象

| 对象 | 说明 |
|------|------|
| 出库通知 `stockoutrequest` | 来源：销售出库申请；含销售明细、数量、仓库等。 |
| 拣货任务 `pickingtask` | 与出库通知关联；**任务壳**，下挂多条 `pickingtaskitem`。 |
| 拣货明细 `pickingtaskitem` | **一条对应一条 `stockitem`**；保留 **`stock_id`**（= 该 `stockitem` 所属汇总桶 `StockAggregateId`），便于校验与展示。 |
| 出库单 `stockout` / `stockoutitem` | 执行出库成功后生成；**以拣货行为准**扣减库存。 |

### 2.2 拣货任务形态：「任务壳 + 仅 stockitem」

- **不再**依赖「仅生成汇总桶级、由系统在桶内自动 FIFO 拆行」的旧生成逻辑作为主路径。  
- **生成拣货**：创建 **`pickingtask` 任务壳**（任务号、仓库、关联出库通知等）；**明细行**由后续「保存拣货结果」写入/覆盖为 **仅含 `stockitem` 维度** 的多行（每行 `stock_id` + `stock_item_id` + 数量等）。

### 2.3 待拣货列表与 FIFO

- **候选列表**：在规则允许范围内列出可选 **`stockitem`**（及展示字段：库位、批次、可用量、汇总桶等）。  
- **FIFO**：**仅用于候选列表默认排序**（建议优先的排在上方），**不**作为系统强制分配顺序。  
- **人工确认**：用户勾选/录入 **若干条 `stockitem` 及数量**；从 **2 条 `stockitem` 凑数** 则 **2 条 `pickingtaskitem`**。

### 2.4 数量与「完成拣货」

- **拣货行数量之和** 必须与 **出库通知应出数量** **严格相等**，否则：  
  - **不允许**点击「完成拣货」。  
- **完成拣货后**：`pickingtask` 状态为已完成（如 `status = 100`），**不允许**再增删改 `pickingtaskitem`。

### 2.5 取消与异常

- **暂不允许**业务侧「取消拣货任务」能力（首版）。  
- **误操作或异常**：接受由**管理员**在库内或管理端修正/解锁类处理（后续可单独立项「管理员作废拣货 / 重开拣货」等）。

### 2.6 执行出库

- **前置**：存在关联该出库通知的拣货任务，且状态为**已完成**；所有拣货行 **`stock_item_id` 非空**（新流程）。  
- **扣账**：**仅**按 `pickingtaskitem` 逐行：`stock_item_id` + `picked_qty`（及 `stock_id`、仓库、销售行等业务校验）扣减 **`stockitem` / 汇总 `stock`**，生成 **`stockoutitem`**，并写入 **`picking_task_item_id`**。  
- **`request.Items`**：**仅作校验**（如按物料汇总与拣货汇总一致）；**以拣货行为准**；不一致则拒绝执行。  
- **`stockout.picking_task_id`**：写入本次对应的 **`pickingtask.Id`**。  
- **不**对 `picking_task_id` 增加数据库 UNIQUE 及额外业务强校验（按已确认口径）。  
- **并发**：首版**不**单独处理并发；以执行时刻读到的已冻结拣货数据为准。

### 2.7 与「申请出库 / 在库可用」

- 出库通知、采购门闸、申请出库时的 **`availableStockQty`** 等**前序规则**可维持现有实现；本方案聚焦 **拣货 → 执行出库** 链路与溯源字段。  
- 若后续希望「申请阶段」与「拣货候选」在备货口径上完全一致，见另文《备货库存可拣货》。

## 三、数据模型变更

### 3.1 `stockout`

| 字段 | 说明 |
|------|------|
| `picking_task_id` | `varchar(36)` NULL；新流程执行成功后写入，指向 `pickingtask.Id`。不加强制 UNIQUE。 |

可选：执行时写入 `PickedTime`（拣货完成时间）、`PickerId`（拣货人用户 Id，与 `pickingtask.OperatorId` 或界面一致）。

### 3.2 `pickingtaskitem`

| 字段 | 说明 |
|------|------|
| `stock_item_id`（列名与项目规范一致） | 非空为**新流程**；历史 NULL **不兼容**新执行。 |
| `stock_id` | **保留**；等于所选 `stockitem` 的 **`StockAggregateId`**。 |

### 3.3 `stockoutitem`（已确认采纳）

| 字段 | 说明 |
|------|------|
| `picking_task_item_id` | 指向 `pickingtaskitem.Id`，便于审计「出库行 ← 拣货行 ← stockitem」。 |

## 四、实施方案

### 4.1 接口划分（建议）

1. **生成拣货任务**：创建 `pickingtask` 壳（关联 `stock_out_request_id`、仓库、操作者等）；**不**再自动生成旧版桶级多行（或仅写占位策略由团队定，默认无行）。  
2. **查询待拣货候选 `stockitem` 列表**：出库通知 + 仓库 + 销售行等过滤；FIFO **仅排序**；返回前端勾选。  
3. **保存拣货结果**（未完成前可多次）：整体替换或增量保存 `pickingtaskitem`；校验每行 `stock_item_id`、`stock_id` 一致性与可用量；**汇总数量 = 出库通知数量**。  
4. **完成拣货**：校验数量已齐 → 置任务完成 → **冻结**明细，拒绝后续修改。  
5. **执行出库**：读冻结拣货行扣账；写 `stockout.picking_task_id`、`stockoutitem.picking_task_item_id`；`request.Items` 仅校验。

### 4.2 服务端核心改造点

| 模块 | 动作 |
|------|------|
| `InventoryCenterService.GeneratePickingTaskAsync` | 改为「任务壳」或与「保存拣货结果」合并流程，**废弃**旧的全自动桶级拆行逻辑（新流程）。 |
| 新增/调整 | 候选 `stockitem` 查询、保存拣货明细、`CompletePickingTaskAsync` 冻结与数量校验。 |
| `StockOutService.ExecuteStockOutAsync` | **主路径**改为遍历 `pickingtaskitem`；删除或禁用旧 FIFO 主路径；历史 NULL `stock_item_id` 直接报错。 |

### 4.3 前端

- 拣货页：展示候选列表（FIFO 排序）→ 勾选/填数 → 保存 → 齐套后「完成拣货」；完成后只读。  
- 执行出库：仍可展示 `request.Items`，提交前与后端校验一致；**扣账以服务端拣货为准**。

### 4.4 迁移与发布

1. 迁移脚本：增加 `stockout.picking_task_id`、`pickingtaskitem.stock_item_id`、`stockoutitem.picking_task_item_id`（可空 → 新数据填全）。  
2. 发布说明：**旧拣货无 `stock_item_id` 不得执行新出库**；需管理员处理或数据修复后重拣货。  
3. 测试用例：单条/多条 `stockitem`、数量不齐禁止完成、完成后禁止改行、执行写溯源字段、重复执行拦截。
