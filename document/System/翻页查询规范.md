# 翻页查询规范（采购订单 / 销售订单 / 需求 RFQ / 报价 / 客户管理 / 供应商管理 / 采购申请 / 入库管理 / 库存管理 / 出库管理 / 付款管理 / 收款管理 / 系统日志）

本文档以当前代码实现为准，归纳 **采购订单**、**销售订单**、**需求（RFQ）**、**报价主表**、**客户主表及回收站/黑名单/冻结列表**、**供应商主表及回收站/黑名单/冻结列表**、**采购申请主列表**、**入库管理（到货通知 / 质检 / 入库单 / 入库批次）**、**库存管理（库存总览分页 / 全库库存明细 / 盘点计划 / 报关移库单）**、**出库管理（出库单 / 出库通知 / 出库明细行）**、**付款管理（付款单 / 进项发票）**、**收款管理（收款单 / 销项发票）**、**系统日志（登录日志 / 操作日志）** 等主表与明细列表的后端分页方式，作为同类列表接口的参考规范。

---

## 1. 通用约定

### 1.1 HTTP 与鉴权

- **基路径**：`/api/v1/purchase-orders`
- **鉴权**：列表类接口需具备 `purchase-order.read`（以 `PurchaseOrdersController` 上特性为准）。
- **当前用户**：从 JWT `ClaimTypes.NameIdentifier` 取用户 ID，写入查询请求的 `CurrentUserId`，用于 **数据范围**（采购订单数据权限）。

### 1.2 查询参数命名（QueryString）

- **`page`**：当前页码，**从 1 开始**；非法或小于 1 时规范化为 **1**。
- **`pageSize`**：每页条数；小于 1 时规范化为 **20**；超过模块上限时 **截断到该模块 `MaxPageSize`**（见下文两表对比）。

### 1.3 服务端分页实现要点

1. 使用 **数据库侧** `Count` + `Skip`/`Take`（EF `IQueryable`），**不在内存中**对全量主表/全量明细做分页。
2. **数据权限**在构建 `IQueryable` 时通过 `IDataPermissionService.ApplyPurchaseOrderDataScopeAsync` 作用在 **采购订单主表** 上；明细列表通过 **明细 JOIN 已收窄的 `PurchaseOrder`** 继承同一范围。付款单 / 进项发票列表见 **§16**；收款单 / 销项发票列表见 **§17**（`ApplyFinanceReceiptListDataScopeAsync` / `ApplyFinanceSellInvoiceListDataScopeAsync`）。系统日志主列表见 **§18**（无业务数据范围，仅 RBAC `rbac.manage`）。
3. **日期筛选**（主单 `CreateTime`）：
   - **起**：`CreateTime >= StartDate`（含起日）。
   - **止**：`CreateTime <= EndDate.AddDays(1)`（与主单列表一致，**含止日全天**的常用写法）。

### 1.4 HTTP 响应体（与前端 axios 拦截器配合）

控制器返回统一外壳，业务数据在 **`data`** 内（前端拦截器会解包 `data`，调用方直接拿到内层对象）：

```json
{
  "success": true,
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "pageSize": 20
  }
}
```

- **`total`**：满足筛选条件的总记录数（**非当前页条数**）。
- **`page`**：当前页（与后端规范化后的页码一致）。
- **`pageSize`**：实际采用的每页条数（可能被上限截断）。

采购订单主表列表在 `data` 中另有 **`aggregates`**（全量筛选维度统计），见下节。

### 1.5 领域层分页结果类型

查询层返回 `PagedResult<T>`，字段与控制器映射关系：

| `PagedResult<T>` | 对外 JSON `data` 字段 |
|------------------|-------------------------|
| `Items`          | `items`                 |
| `TotalCount`     | `total`                 |
| `PageIndex`      | `page`                  |
| `PageSize`       | `pageSize`              |

| **默认 `pageSize`** | 20（小于 1 时按 20） |
| **排序** | **主单 `CreateTime` 降序**，同主单内 **`PurchaseOrderItemCode` 升序**（`OrderByDescending(po.CreateTime).ThenBy(item.PurchaseOrderItemCode)`） |
| **行模型** | 查询层为 `PurchaseOrderItemListLineRaw`；API 层映射为匿名对象并做掩码、**创建人登录名补全**等 |

### 3.1 查询参数

| Query 参数 | 说明 |
|------------|------|
| `startDate` / `endDate` | 按 **主单** `CreateTime`，规则同 §1.3。 |
| `purchaseOrderCode` | 主单采购单号包含。 |
| `vendorName` | 主单供应商名称包含；**无供应商信息权限时控制器清空该条件**（防越权筛参）。 |
| `purchaseUserName` | 主单采购员名称包含；**无采购员读权限时控制器清空该条件**。 |
| `pn` | 明细料号包含。 |
| `orderType` | 主表类型。 |
| `page` / `pageSize` | 分页；`pageSize` 上限 **100**。 |

### 3.2 数据与权限

- **JOIN**：`PurchaseOrderItem` **inner join** 数据权限已过滤的 `PurchaseOrder`；**left join** `PurchaseOrderItemExtend` 取扩展进度与已请款金额等。
- **创建人**：批量查 `Users.UserName`，输出字段如 **`createUserName`**（与列表列展示约定一致）。

### 3.3 路由设计注意（已实现）

子资源路径 **`items`** 为字面段，为避免与 **`GET .../purchase-orders/{id}`** 冲突，主单按 ID 访问的路由应使用 **`{id:guid}`** 等约束，保证 **`items` 不会被误解析为订单 ID**。

## 4. 两表分页差异一览

| 维度 | 采购订单主表 `GET /api/v1/purchase-orders` | 采购订单明细 `GET /api/v1/purchase-orders/items` |
|------|---------------------------------------------|--------------------------------------------------|
| 分页对象 | 主表一行一单 | 明细一行一条 |
| `MaxPageSize` | **2000** | **100** |
| 稳定排序键 | 主单 `CreateTime` DESC | 主单 `CreateTime` DESC + 明细 `PurchaseOrderItemCode` ASC |
| 典型额外负载 | `aggregates` 全量统计 | 行级掩码 + 创建人登录名补全 |
| 大批量导出心智 | 允许较大 `pageSize`（仍受 2000  cap） | **不**承担「单请求扫全表明细」；需缩小筛选或多次分页 |

---

## 5. 新增类似列表时的建议 checklist

1. 在 **Infrastructure** 专用 Query 类中集中：`BuildQuery` → `CountAsync` → `OrderBy` → `Skip`/`Take` → `ProjectTo DTO/Raw`。
2. 显式声明 **`MaxPageSize`** 常量并在文档/PRD 中与产品对齐。
3. **排序**写死或白名单映射，禁止未校验的 `sort` 字符串直接拼 SQL。
4. 列表响应统一 **`items` / `total` / `page` / `pageSize`**，与现有 axios 解包约定一致。
5. 若存在 **`report`、`items` 等字面路由**，注意与 **`{id}`** 路由顺序及 **路由约束** 避免歧义。

## 18. 系统日志（主菜单「系统日志」板块）

以下两个列表在 **`log_login` / `log_operation`** 上使用 EF **`CountAsync` + `Skip`/`Take`**（由 `ApplicationDbContext` 映射），**不在内存中**拉全表再分页。接口外层为 `ApiResponse<T>`，业务体为 **`LoginLogPagedResult` / `OperationLogPagedResult`**（字段 **`Total`**、**`Page`**、**`PageSize`**、**`Items`**，与 §1.4 的 `total`/`page`/`pageSize`/`items` 命名不同，以前端 `loginLogs.ts` / `operationLogs.ts` 类型为准）。

### 18.1 登录日志列表

| 项目 | 约定 |
|------|------|
| **GET** | `/api/v1/login-logs` |
| **实现** | `CRM.Infrastructure/SystemLogs/LoginLogListQuery.cs`（实现 `ILoginLogQueryService`，`MaxPageSize = 2000`） |
| **注册** | `CRM.Infrastructure/Extensions/ServiceCollectionExtensions.cs`（`AddInfrastructure` 内 `AddScoped`） |
| **Query** | **`page`**、**`pageSize`**（默认 20）；**`userId`**（精确匹配）；**`loginAtFrom`** / **`loginAtTo`**（`LoginAt` 时间范围，UTC 规范化与旧版一致） |
| **排序** | **`LoginAt` 降序** |
| **前端列表** | `CRM.Web/src/views/System/LoginLogList.vue`、`CRM.Web/src/api/loginLogs.ts` |

### 18.2 操作日志列表
