# 销售订单使用采购备货 — 业务逻辑说明

## 1. 文档目的

说明：当存在 **采购备货**（备货采购入库形成的 **备货类型库存**）时，系统如何在 **销售订单明细** 侧体现「当前还有多少备货库存可用」，以及该数据在哪些 **业务节点** 被刷新、**前端如何消费**该字段。

核心载体为销售明细扩展表字段 **`PurchasedStock_AvailableQty`**（API / 前端 JSON 多为 **`purchasedStockAvailableQty`**）。

---

## 2. 概念对齐

### 2.1 采购备货（本业务所指）

- **备货采购订单**：`PurchaseOrder.Type = 2`（备货采购），用于补充备货池，通常不绑定具体销售订单行。
- **备货库存**：库存主档 `StockInfo` 中 **`StockType` 为备货**（实现中与 `StockInventoryTypeCodes.Stocking` 一致），使用 **`PurchasePn` / `PurchaseBrand`** 与物料维度对齐。

### 2.2 销售侧如何「看到」备货可用量

- 销售明细 **`SellOrderItem`** 与扩展表 **`SellOrderItemExtend`** 一对一（主键同为销售明细 Id）。
- **`PurchasedStock_AvailableQty`**：与销售明细的 **`PN` + `Brand`** 相匹配（忽略大小写、首尾空白）时，**全库所有备货类型库存行** 的 **`QtyRepertoryAvailable` 之和**，再 **截断为非负整数**。
- **重要**：该值为 **同 PN+Brand 下备货池的总量快照**，**不是**本销售行独占可出数量，也 **不与销售行数量比较**；同一 PN+Brand 的多条未完结销售明细，通常会显示 **相同** 的合计值。

### 2.3 哪些销售行会被更新

对某一 `(PN, Brand)` 重算时，会更新所有满足以下条件的销售明细扩展：

- 销售明细 **未取消**、数量 **> 0**；
- `PN`、`Brand` 与当前键一致；
- 存在扩展行；
- **`StockOutProgressStatus` 未为出库完成**（完成后不再维护该快照）。

---

## 3. 三个业务刷新节点（与实现对齐）

以下三处触发 **`SellOrderItemPurchasedStockAvailableSyncService`** 对受影响 **`(PN, Brand)`** 的 **`RecalculateByPurchasePnAndBrandAsync`**（或经 `Try*` 方法等价路径）。

### 节点 ①：采购入库过账完成（入库单 `Status == 2`）

- **位置**：`StockInService`，入库单状态变为 **已完成（2）** 的处理链中（过账、物流回写等之后）。
- **调用**：`TryRecalculateFromCompletedStockInAsync(StockIn)`。
- **规则**：仅当关联采购订单类型为 **备货采购（备货）** 时，从入库明细解析 **采购明细 PN、Brand**，加入重算键并刷新相关销售扩展。
- **含义**：备货采购 **入库完成并过账** 后，备货池增加，销售侧需同步 **备货可用量合计**。

### 节点 ②：销售出库执行完成（扣减备货库存）

- **位置**：`StockOutService.ExecuteStockOut`，在库存变更持久化、记录出库之后。
- **调用**：`TryRecalculateFromChangedStockInfosAsync(changedStocks)`。
- **规则**：仅处理本次变更中 **`StockType` 为备货** 的 `StockInfo`，按 **`PurchasePn` / `PurchaseBrand`** 去重后重算。
- **含义**：销售出库 **减少备货在库可用量** 后，同 PN+Brand 下仍维护该字段的销售明细看到 **更新后的池总量**。

### 节点 ③：销售订单明细新建或整单替换明细

- **位置**：`SalesOrderService` 的 **`CreateAsync`**、**`UpdateAsync`**（携带新明细列表并删旧建新时）。
- **时机**：每条新明细 **`SellOrderItemExtendSyncService.RecalculateAsync`** 之后，对本次行的 **`(PN, Brand)` 去重** 后逐个 **`RecalculateByPurchasePnAndBrandAsync`**。
- **含义**：覆盖 **「先备货入库、后建销售订单」** 场景；若仅有节点 ①、②，后建单的扩展行可能长期为默认 0，建单/换行时主动对齐当前备货池。

### 异常与一致性

- 节点 ①、②、③ 对备货可用量同步失败时 **记日志**，**不阻断** 主流程（建单/入库/出库）。
- 若库存通过 **未纳入上述路径** 的方式变更，可能出现短暂不同步；主路径以三节点为准。

## 4. 前端对 `purchasedStockAvailableQty > 0` 的用途

实现位置：**`CRM.Web/src/constants/salesOrderStatus.ts`**。

1. **`salesOrderLinePurchasedStockReliefOk(row)`**  
   - 当 **`Number(row.purchasedStockAvailableQty)` 为有限数且大于 0** 时返回 `true`。  
   - 语义：**同 PN+品牌的备货在库可用量合计大于 0**，认为存在可依赖的备货池。

2. **与「申请出库」按钮、操作入口的关系**  
   - **`salesOrderLineApplyStockOutButtonDisabled`**：若出库进度已为完成（2），一律禁用；否则若 **`salesOrderLinePurchasedStockReliefOk(row)`** 为真，则 **不禁用**（即 **不因**「待采购 / 未过采购门槛」单独锁死按钮）。  
   - 列表 / 详情（如 **`SalesOrderItemList.vue`**、**`SalesOrderDetail.vue`**）在判断是否允许申请出库时，采用与 **`stockOutApplyPurchaseGateOk`** 并列的逻辑：**采购门槛通过** 或 **备货可用量放宽通过** 之一即可继续相关操作。

3. **产品表述（给业务与测试）**  
   - **当扩展表返回的备货可用量合计 > 0 时，允许在尚未满足原「须先采购」类门槛的情况下，仍可对该行发起/使用申请出库相关能力**（具体 UI 以页面为准）。  
   - 该放宽 **不表示** 本行已锁定等量库存；出库仍受实际拣货与库存分配约束。

## 5. 模型注释与库列注释（权威文案）

变更时请 **同步** 模型 XML、库 `COMMENT ON COLUMN`（迁移）与 **本节文字**。

### 5.1 C# 模型（`SellOrderItemExtend.PurchasedStock_AvailableQty`）

文件：`CRM.Core/Models/Sales/SellOrderItemExtend.cs`。

摘要含义（与源码 `<summary>` 一致）：

- 同 PN+品牌下所有备货库存（`StockType`=备货）在库可用量 `QtyRepertoryAvailable` 之和（截断为非负 int；不与销售数量比较；同键多销售行共享同一池合计）。
- **刷新**：采购备货入库单状态为已完成(2)过账后；销售出库执行扣减备货库存后；销售订单明细新建或整单替换明细后。
- **前端**：API 字段 `purchasedStockAvailableQty` 大于 0 时在申请出库场景放宽「须先满足采购门槛」的限制（见 `salesOrderLinePurchasedStockReliefOk`）。

### 5.2 PostgreSQL 列注释（`sellorderitemextend."PurchasedStock_AvailableQty"`）

由迁移 **`20260510130000_SellOrderItemExtendPurchasedStockAvailableQtyComment`**（紧接列添加之后）更新后的 `COMMENT ON COLUMN` 全文为：

> 同PN+品牌下备货库存(StockType=2)在库可用量之和；入库单过账完成(备货采购)、销售出库扣减备货后、销售明细新建或整单替换后重算；前端大于0时放宽申请出库的采购门槛限制

（列首次添加见 **`20260510120000_SellOrderItemExtendPurchasedStockAvailableQty`**。）
