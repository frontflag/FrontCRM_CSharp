# 业务规则总览

本文档根据 **`CRM.Core` 服务层** 当前实现梳理业务规则，便于产品、测试与二次开发对齐。**以前端展示为辅、以后端校验与持久化为准**；未在服务端落地的约定不在此列。

**维护说明**

- 实现位置以各节标注的类名为准。
- `PurchaseRequisitionService.cs` 中 `#if false` … `#endif` 内为历史重复代码，**不参与编译**，勿与文件前部活动实现混淆。
- 财务状态机细节可与 [财务 Phase1 状态流转矩阵](../PRD/辅助功能/财务Phase1状态流转矩阵.md) 对照；销售主状态枚举见 [销售订单主状态](./销售订单主状态.md)（**数值请以代码枚举 `SellOrderMainStatus` 为准**）。

## 1. 主数据：客户 / 供应商

| 规则 | 说明 | 实现 |
|------|------|------|
| 建档与导入 | 客户/供应商名称非空；导入时联系人需有姓名 | `CustomerService`、`VendorService` |
| 主状态机 | `1新建→2待审核→10已审核`；`2` 可至 `10` 或 `-1审核失败`；`-1` 可回到 `1`；`10→12待财务审核→20财务建档`（`10` 也可直接 `20`，以客户/供应商各自 `Validate*StatusTransition` 为准） | `ValidateCustomerStatusTransition` / `ValidateVendorStatusTransition` |
| 审核拒绝 | 目标状态为审核失败时**必须**填写原因 | `UpdateStatusAsync` |
| 黑名单 | 移出黑名单须填原因 | `RemoveFromBlackListAsync` |
| 冻结 | 冻结须填原因；已冻结不可再冻；未冻结不可「启用」 | `FreezeCustomerAsync` 等 |

## 2. 数据权限（列表/查询范围）

| 规则 | 说明 | 实现 |
|------|------|------|
| 系统管理员 | 不按数据范围过滤 | `DataPermissionService`（`IsSysAdmin`） |
| 销售数据范围 | `0` 全部、`1` 本人（`SalesUserId`）、`2`/`3` 部门（含子部门由 `includeChildren` 区分）、`4` 无权限 | `FilterCustomersAsync`、`FilterSalesOrdersAsync`、RFQ 销售侧等 |
| 采购数据范围（采购订单等） | `0`/`1`/`2`/`3`/`4` 与客户侧对称，单据字段为 **`PurchaseOrder.PurchaseUserId`**；RFQ 采购侧可按明细 **`AssignedPurchaserUserId1/2`** 与范围匹配 | `FilterPurchaseOrdersAsync`、`FilterRFQsAsync` 等 |
| 供应商主数据列表 | **不按** `VendorInfo.PurchaseUserId` 缩小：仅 **`PurchaseDataScope == 4`** 时列表为空，否则**全部供应商**（报价等任选；专属供应商规则待模型字段） | `FilterVendorsAsync`、`CanAccessVendorAsync` |
| 对供应商财务单据 | 经 **`VendorId` → Vendor.PurchaseUserId`** 再套 **`PurchaseDataScope`**；财务部不按供应商采购员缩小 | `FilterFinancePaymentsAsync`、`FilterFinancePurchaseInvoicesAsync` |
| 财务单据（销售侧） | 收款等另有 `FilterFinance*` 方法 | 同文件 |

详见 [用户角色与数据隔离实现文档](../EBS/用户角色与数据隔离实现文档.md)（偏历史 EBS 对照）。**业务员—客户与采购员—供应商**对照说明见 [数据权限-业务员客户与采购员供应商](./权限/数据权限-业务员客户与采购员供应商.md)。

## 3. 需求（RFQ）

| 规则 | 说明 | 实现 |
|------|------|------|
| 创建 | 必选客户；至少一条明细；每条明细 **MPN**、**品牌** 非空（品牌经 `NormalizeLineString`） | `RFQService.CreateAsync` |
| 单号 | 服务端生成（流水号模块 `ModuleCodes.RFQ`） | `ISerialNumberService` |
| 采购员轮询 | 新建/更新并重发明细时，从系统参数与角色解析采购员池，取**连续 2 人**写入该单全部明细的 `AssignedPurchaserUserId1/2`，全局游标 `+2`；池为空则状态可为待分配、`AssignMethod` 等按代码分支 | `TakeNextRoundRobinPurchaserPairAsync` 等 |
| 手工分配 | 关闭/取消状态（`7`/`8`）不可分配；采购员须存在且 `IsActive`；写入后明细统一为该采购员，`AssignMethod=4` | `AssignPurchaserAsync` |
| 状态更新 | `UpdateStatusAsync` **直接写库**，无状态机校验 | `RFQService.UpdateStatusAsync` |

## 4. 报价（Quote）

| 规则 | 说明 | 实现 |
|------|------|------|
| 单号 | **忽略客户端**，服务端 `ModuleCodes.Quotation` 生成 | `QuoteService.CreateAsync` |
| 与需求明细绑定 | 以 **`RFQItemId`** 为准；禁止仅用 `RFQId+Mpn` 推断（同一需求可多条相同 MPN） | 注释与实现 |
| 回写需求明细状态 | 创建报价后，若带有效 `RFQItemId` 且该明细当前状态为 **0（待报价）**，则改为 **1（已报价）**；否则不覆盖 | `QuoteService.CreateAsync` |

## 5. 销售订单

| 规则 | 说明 | 实现 |
|------|------|------|
| 创建 | 客户 ID 必填；明细逐行落库；主表 `Total`/`ConvertTotal` = Σ(数量×单价)；明细扩展表 `SellOrderItemExtend` 按行小计初始化开票/收款等「未核销」额度 | `SalesOrderService.CreateAsync`、`AddSellOrderItemExtendAsync` |
| 主状态枚举 | `SellOrderMainStatus`：`New=1`、`PendingAudit=2`、`Approved=10`、`InProgress=20`、`Completed=100`、`AuditFailed=-1`、`Cancelled=-2` | `SellOrderMainStatus.cs`、`SalesOrderService.UpdateStatusAsync` |
| 状态流转 | 仅 `New→PendingAudit`；`PendingAudit→Approved/AuditFailed`；审核失败可带 `AuditRemark` | `UpdateStatusAsync` |
| 申请出库（主单） | `RequestStockOutAsync` 将主状态更新为 **InProgress（20）** | `SalesOrderService` |

## 6. 采购订单

| 规则 | 说明 | 实现 |
|------|------|------|
| 创建 | 供应商必填；至少一行明细；主单总额 = Σ(数量×成本)；明细扩展 `PurchaseOrderItemExtend` 初始化入库通知/发票/付款等额度 | `PurchaseOrderService.CreateAsync` |
| 主状态（数值） | `1新建→2待审→10已通过→20待确认→30已确认→50进行中→100完成`；异常：`-1审核失败`、`-2取消` | `PurchaseOrderService` 常量与 `ValidateStatusTransition` |
| 状态同步明细 | 当目标状态为 `1/2/10/20/30` 时，明细 `Status` 与主单同步 | `ShouldSyncOrderAndItemStatus` |
| 从销售自动生成 | 仅处理 `SellOrderItem.Status==0` 的明细；按行生成简化 PO（供应商占位等） | `AutoGenerateFromSellOrderAsync` |

## 7. 采购申请（Purchase Requisition）

**活动实现**为 `PurchaseRequisitionService.cs` 文件**前半段**（至第一个 `#if false` 之前）。

| 规则 | 说明 |
|------|------|
| 可申请数量口径 | `purchasedQty` = 同一销售明细下 **采购订单明细数量**之和；`openPurchaseRequisitionQty` = 采购申请状态 **0 或 1** 的数量之和；**3=已取消**不计入占用；**2=完成** 不计入 open（由已下采购体现）；`remainingQty = 销售明细数量 − purchasedQty − openPR`；`remainingQty≤0` 的不出现在行选项中。 |
| 多次申请 | **同一销售明细允许**多次创建采购申请，只要每次 `Qty≤remainingQty` 且行仍在选项集合中（服务端创建时再次校验）。 |
| 创建校验 | 销售明细须存在；`Qty>0`；`Qty≤line.remainingQty`。 |
| 报价回填 | 若销售明细带 `QuoteId`，**仅**按该 ID 加载报价主表下的报价明细列表，再按 `CreateTime`、明细主键 `Id` 稳定取**一条**用于默认供应商与 `QuoteCost`；**不再**用 PN/品牌/币别/单价/DateCode 与销单行逐字段匹配。实现：`QuoteItemForPrResolver`、`PurchaseRequisitionService.CreateAsync`、`PurchaseRequisitionsController.GetById`。 |

**说明**：当前活动实现中 **未** 校验销售订单是否已审核；若业务要求「仅已审订单可请购」，需在 `CreateAsync` 中补充与 `StockOutService` 类似的主单状态校验。

## 8. 出库通知与执行出库

| 规则 | 说明 | 实现 |
|------|------|------|
| 申请前提 | 销售订单存在；主单 `Status≥Approved(10)` 且 **非** `Completed(100)`；明细属于该单且 **未取消**（`SellOrderItem.Status==0`） | `StockOutService.CreateRequestAsync` |
| 多次申请 | 对同一 `SalesOrderItemId`，汇总 `StockOutRequest.Quantity`（**`Status≠2` 视为占用**，`2` 视为已取消不占额度）；`Σ≤销售数量`，否则拒绝。 |
| 单号 | `RequestCode` 空则服务端 `ModuleCodes.StockOutRequest` 生成。 | 同左 |
| 申请上下文 API | 返回订单数量、已占用、剩余可申请、库存可用等，供前端**仅展示**（`suggestedMaxQty = min(剩余, 可用)`）。 | `GetApplyContextAsync` |
| 执行出库 | 必须关联出库申请；申请状态已执行(1)/已取消(2) 不可再执行；须存在拣货任务且**至少一条状态为 100（完成）**；按明细扣减库存 | `ExecuteStockOutAsync` |

## 12. 编码与流水号

业务单号（销售订单、采购订单、RFQ、报价、入库、出库申请、财务单据等）一般由 **`ISerialNumberService` + `ModuleCodes`** 在服务端生成，客户端传入常被忽略或仅作占位。细则见 [业务流水号编码规范](../PRD/规范/业务规范/业务流水号编码规范.md)。

## 13. 横切：销售明细 × 采购 / 付款 / 出库

在**当前后端实现**下可归纳为：

| 维度 | 是否支持「一条销售明细多次」 | 约束要点 |
|------|------------------------------|----------|
| 采购申请 | 是 | 总申请占用受 `remainingQty` 与 open PR 口径约束 |
| 采购订单明细 | 是（多 PO 行可指向同一 `SellOrderItemId`） | 与 PR、请购数量合计在业务上需一致 |
| 付款申请 | 是（针对 **采购订单明细** 维度多次请款/核销） | `FinancePaymentService` 按 PO 行聚合核销 |
| 出库通知 | 是 | 非取消通知数量累计不超过销售行数量 |

## 14. 销售毛利 / 利润口径（产品约定，与实现逐步对齐）

以下为客户确认的口径，供利润类功能与报表实现时遵循；**不单独依赖**一套与业务单据脱钩的「财务模块公式」。

| 主题 | 约定 |
|------|------|
| **报价利润** | 销售明细与报价的关联**只**通过 `SellOrderItem.QuoteId` 与**报价主表**对应；报价侧成本行与采购申请回填一致（见 §7「报价回填」）。无 `QuoteId` 时报价利润为 **null 或 0**。 |
| **采购利润 / 利润率** | 用 **销售订单明细的销售单价×销售数量** 与 **采购订单明细的采购单价×采购数量**（通过 `SellOrderItemId` 等关联）计算即可，无需再套一层独立财务公式。 |
| **预计销售利润（首张已确认采购单）** | 「第 1 张」采购单按 **`PurchaseOrder` 的确认时间**先后排序。**建议**主表增加专用确认时间字段并在状态变为已确认(30)时写入；在字段落地前，实现可与产品约定临时口径（例如该次流转的 `ModifyTime`），并在代码注释中标明。 |
| **出库利润 · 批次成本** | 出库消耗批次对应的成本以**库存记录上的采购单价**为准；该单价来自**本批次入库**所关联的**入库通知**对应的**采购订单明细**上的采购单价、币别及 **美金折算单价（`convert_price`）**。实现出库利润时需与 `StockInService` / 库存台账回写及 `StockOutService` 取价逻辑保持一致。 |
