# 列表「复选篮子」批量操作规范 PRD（V1.0）

## 1. 文档目的

统一系统中**跨多条列表记录、且需分页或长时间勾选后一并处理**的批量操作体验，避免：

- 翻页后勾选丢失，用户误以为未选中
- 仅依赖表格当前页勾选，与「批量」心智不一致
- 各列表页批量交互形态分散，增加学习与维护成本

本规范将需求明细页已落地的「复选篮子」提炼为**全项目标准模式**。

---

## 2. 适用范围（强制）

凡满足以下任一条件的列表页，**必须**采用本规范的「复选篮子」能力（含 UI 布局、跨页状态、查看/清空路径），**禁止**仅使用「当前页勾选 + 立即批量」而无篮子沉淀：

1. 列表存在**分页**，且用户可能**在多页勾选**后执行同一批量动作（如批量报价、批量提交、批量导出、批量删除草稿等）。
2. 批量操作前需要用户**确认已选集合**（条数、关键字段），或允许在提交前**移除单条**。
3. 业务上明确为「先选多条、再统一处理」的列表批量场景。

**不适用**（可选用普通多选，不必单独篮子 Store）：

- 单页内全选即操作、且列表无分页或不允许跨页意图的极简场景（仍建议与产品确认后再豁免）。
- 非列表结构的多选（如树、穿梭框）按其专用组件规范处理。

---

## 3. 术语

| 术语 | 说明 |
|------|------|
| **复选篮子** | 与用户勾选同步的**跨页持久集合**（通常用 Pinia 维护 id → 行快照 + 顺序）。 |
| **当前页合并** | 用户在本页勾选变化时，只增删「本页 id」在篮子中的成员，**其它页已选记录保持不变**。 |
| **批量操作** | 依赖篮子内全部（或筛选后）记录执行的业务动作，如批量报价、批量审核等。 |

---

## 4. 交互与布局规范（强制）

### 4.1 表格

1. 使用行多选列：`type="selection"`，且 **`reserve-selection="true`**（与 `row-key` 配合），减少翻页时勾选闪烁与状态错乱。
2. 在 `@selection-change` 中调用篮子的 **「按当前页合并」** 逻辑（见第 6 节），**不要**用「每次全量替换篮子 = 当前页选中行」的简单实现。

### 4.2 页脚工具区（列表表格与下方区块之间）

1. **左侧**：固定放置一组控件（与需求明细页一致）：
   - **复选篮子**：`el-button` **`link` + `type="primary"`**，文案为「复选篮子」，有记录时显示数量「（n）」，数量可用强调色样式与正文区分。
   - **清空篮子**：与「复选篮子」**同款 label 链式按钮**（`link` + `type="warning"`），文案「清空篮子」；**仅当 n > 0 时显示**。
2. 两个控件**横向紧挨**（小间距 `gap`，如 2px～8px 量级），**左对齐**于表格内容区域左侧（与列表字段起点对齐）。
3. **右侧**：分页等其它控件仍靠右（如 `margin-left: auto`），与左侧篮子组同一行，窄屏允许换行。

### 4.3 抽屉（查看篮子）

1. 点击「复选篮子」打开 **`el-drawer`**（建议 `direction="rtl"`），标题「复选篮子」。
2. **空状态**：说明如何向篮子添加记录（勾选行、翻页保留等）。
3. **有数据**：顶部一句说明 + 表格列出篮子内记录；每行提供「移除」；说明句中的「清空篮子」须为与页脚**同款的 `link` + `warning` 按钮**，点击执行与页脚相同的清空流程（含确认框等业务规则时保持一致）。
4. 抽屉样式若挂载在 `body`，需在非 scoped 样式块中按抽屉根 class 编写皮肤（参考需求明细实现）。

### 4.4 批量主按钮

1. 页面级主批量动作（如「批量报价」）建议放在**页眉右侧**或列表上方工具栏，**依赖篮子条数**：`disabled` 当 `n === 0`。
2. 执行时**以篮子 Store 中的列表为准**，不以「仅当前页勾选」为准。

### 4.5 清空篮子

1. 页脚与抽屉内清空均应：**二次确认**（`ElMessageBox`）后再 `clear()`，并同步清空表格 `clearSelection()`（若需要），避免 UI 与状态不一致。

---

## 5. 状态与数据规范

1. 篮子内至少保存：**稳定主键 id**、批量操作与展示所需的**行快照字段**（如编号、客户、物料、数量等）；在 `mergePageSelection` 时用当前表格行**更新快照**，避免展示过期数据。
2. 切换查询条件、重置筛选导致**当前列表数据变化**时，产品需约定是否清空篮子或仅剔除已不在结果集中的 id；**默认建议**：在「查询/重置」时清空篮子或在文案中提示用户，避免批量提交已不可见行（实现按业务选择并在页面注释说明）。

---

## 6. 技术实现要点（推荐）

1. **Pinia Store**（可按列表拆分 Store，或抽象通用 `useListBasketStore<T>()`）：
   - `itemsById: Record<id, Row>` + `idOrder: id[]` 保证顺序与去重。
   - `mergePageSelection(allPageRows, selectedOnPage)`：仅对本页 id 做「选中则 upsert、未选则从篮子 remove」。
   - `upsert` / `remove` / `clear` / `has` / 只读 `count` / `items`。
2. 列表加载后若需恢复勾选：根据 `basketStore.has(id)` 对当前页行 `toggleRowSelection(row, true)`（注意使用 `nextTick` 与防抖合并，避免与 `selection-change` 循环）。
3. 样式类名建议统一前缀，如 `basket-footer-left`、`basket-open-btn`、`basket-clear-btn`，便于主题统一。
