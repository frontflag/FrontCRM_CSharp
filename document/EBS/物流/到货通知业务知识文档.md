# 到货通知业务知识文档

## 一、数据库表结构设计

### 1. 到货通知表（单表结构）

| 表名 | 类型 | 说明 |
|------|------|------|
| **StockInNotify** | 单表 | 到货通知表，无明细表 |

**设计说明**：
- 一条记录代表一个物料的一次到货批次
- 物料信息（PN、Brand）直接存储在主表中
- 粒度已经是最细（按物料+批次），无需明细表

### 2. 关键字段说明

```csharp
public class StockInNotify : EnterpriseBaseEntity
{
    // 主键
    public Guid StockInNotifyId { get; set; }
    
    // 关联字段
    public Guid PurchaseOrderItemId { get; set; }  // 关联采购明细ID
    public Guid SellOrderItemId { get; set; }      // 关联销售明细ID
    public Guid PurchaseOrderId { get; set; }      // 关联采购订单ID
    
    // 核心数量字段
    public long ExpectQty { get; set; }    // 预期到货数量
    public long ReceiveQty { get; set; }   // 实际到货数量（入库后更新）
    
    // 物料信息（直接存储）
    public Guid ProductId { get; set; }
    public string PN { get; set; }         // 型号
    public string Brand { get; set; }      // 品牌
    
    // 价格信息
    public decimal Cost { get; set; }              // 采购单价

    public decimal ReceiveTotal { get; set; }      // 实际总额
    
    // 状态
    public short Status { get; set; }              // 状态（待收货/已收货/质检中）
    
    // 其他字段...
}
```

### 3. 相关表对比

## 二、采购订单与到货通知关系

### 1. 一对多关系

- **一个采购订单明细** 可以生成 **多条到货通知记录**
- 分批次到货时，每次生成新的到货通知记录

### 2. 示例场景

**采购订单明细数量1000，分两次生成到货通知：**

**采购订单明细表 (PurchaseOrderItem)**

| 字段 | 值 |
|------|-----|
| PurchaseOrderItemId | {GUID} |
| Qty | 1000 |

**到货通知表记录：**

| StockInNotifyId | PurchaseOrderItemId | ExpectQty | ReceiveQty | 说明 |
|-----------------|---------------------|-----------|------------|------|
| GUID-1 | {POItemId} | 100 | 0 | 第一次到货通知 |
| GUID-2 | {POItemId} | 200 | 0 | 第二次到货通知 |

**关系说明**：
- 同一采购明细ID对应多条到货通知记录
- 每条记录的 ExpectQty 是本次到货数量，不是累计

## 三、数量计算逻辑

### 1. 核心数量字段

| 字段 | 位置 | 说明 |
|------|------|------|
| **Qty** | PurchaseOrderItem | 采购订单数量（原始订单数量） |
| **ExpectQty** | StockInNotify | 预期到货数量（本次到货数量） |
| **ReceiveQty** | StockInNotify | 实际到货数量（入库后更新） |
| **QtyStockInNotifyNot** | PurchaseOrderItemExtend | 剩余到货通知数 |

### 2. 数量关系公式

```
剩余到货通知数 = 采购数量 - 已到货数 - 在途数

QtyStockInNotifyNot = Qty - ReceiveQty(累计) - ExpectQty(未到货累计)
```

### 3. 示例计算

**采购数量1000，分两次到货通知：**

| 阶段 | QtyStockInNotifyNot | 计算过程 |
|------|---------------------|----------|
| 初始状态 | 1000 | = 1000 - 0 - 0 |
| 第一次到货通知(100)后 | 700 | = 1000 - 0 - 100 - 200 |
| 第二次到货通知(200)后 | 500 | = 1000 - 0 - 300 |
| 第一次入库(100)后 | 600 | = 1000 - 100 - 300 |

## 四、剩余到货通知数字段详解

### 1. 字段定义

| 表名 | 字段名 | 数据类型 | 说明 |
|------|--------|----------|------|
| PurchaseOrderItemExtend | **QtyStockInNotifyNot** | long | 剩余到货通知数 |

```csharp
// PurchaseOrderItemExtend.cs
/// <summary>
/// 剩余到货通知数
/// </summary>
[Description("剩余到货通知数")]
public long QtyStockInNotifyNot { get; set; }
```

### 2. 字段作用

- **避免动态计算**：查询时无需联表计算
- **提高查询效率**：直接读取字段值，适合列表筛选
- **数据一致性**：由业务逻辑保证，统一更新

### 3. 更新时机

#### 3.1 创建采购订单时初始化
```csharp
// PurchaseOrderService.cs
QtyStockInNotifyNot = item.Qty  // 初始值 = 采购数量
```

#### 3.2 入库完成后更新
```csharp
// StockInService.cs - HandleQtyStockInNotifyNot 方法
purchaseOrderItemExtend.QtyStockInNotifyNot = purchaseOrderItem.Qty - stockInNotify.ReceiveQty;
```

#### 3.3 修改采购数量时校验和调整
```csharp
// PurchaseOrderService.cs
if (purchaseOrderItem.Qty < receiveQty + expectQty) {
    throw new BusinessException("采购数量不能小于已添加到货通知数量");
}
```

### 4. 查询使用

```csharp
// 查询还有剩余到货通知待添加的采购明细
query.Where(poe => poe.QtyStockInNotifyNot > 0)

// PurchaseOrderItemService.cs 第964行
where poe.QtyStockInNotifyNot > 0
```

## 五、到货通知生成流程

### 1. 生成到货通知接口

```csharp
// StockInNotifyService.cs
public List<Guid> AddStockInNotifys(List<StockInNotifyRequestDto> stockInList)
```

### 2. 数量校验逻辑

```csharp
// 生成到货通知时的数量校验
var haveNum = 已到货数量(ReceiveQty累加);
var processNum = 在途数量(ExpectQty累加);
var numwrite = purchaseOrderItem.Qty + newExpectQty - haveNum - processNum;

if (param.ExpectQty > numwrite) {
    throw new BusinessException("预期到货数量不能大于剩余可到货数量！");
}
```

### 3. 每次生成新记录

- 每次生成到货通知都会创建新的 `StockInNotify` 记录
- 同一采购明细可对应多条到货通知记录
- 通过 `PurchaseOrderItemId` 关联

## 六、关键业务规则

### 1. 数量约束

| 约束 | 说明 |
|------|------|
| **不能超过剩余数量** | ExpectQty ≤ QtyStockInNotifyNot |
| **修改采购数量限制** | 新数量 ≥ 已到货数 + 在途数 |
| **入库数量限制** | 入库数量 ≤ ExpectQty |

### 2. 状态流转

```
生成到货通知 → 在途/待收货 → 入库完成 → 已收货
     ↑                                        ↓
     └──────────── 质检不合格退货 ────────────┘
```

### 3. 事务机制

- 批量生成到货通知时，单条失败则全部回滚
- 入库操作时同时更新 `QtyStockInNotifyNot` 字段
