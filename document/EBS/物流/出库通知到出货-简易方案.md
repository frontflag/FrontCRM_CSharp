# 出库通知 → 出货：简易业务流程实现方案（MVP）

> 目标：在**不先做完整装箱单 UI** 的前提下，先把「出库通知 → 拣货 →（备货）→ 实物出库/扣账 → 发货信息」**跑通**，后续再按《装箱拣货出库.md》补齐装箱单、审核流、多批次界面等细节。

---

## 一、与《装箱拣货出库.md》的关系

| EBS 文档概念 | 文档中的状态（节选） | 当前 CRM 中的对应（现状） |
|-------------|---------------------|---------------------------|
| 出库通知 StockOutNotify | 通知行状态：新建→部分装箱→已装箱→已出库 | **`StockOutRequest`**（前端「出库通知」列表），`Status`：0 待处理 / 1 已出库 / 2 已取消 |
| 拣货 CheckProduct（15） | 系统选批次、预占、生成装箱明细 | **`PickingTask` + PickingTaskItem**，`CompletePickingTask` 后 `Status=100` |
| 备货 ProductReady（16） | 仓库实物核对后状态前进 | **MVP 可与「完成拣货」合并为一步**，或仅增加**无库存副作用**的状态位（见下文） |
| 出库 / 发货 | 准备出库(19)→已出库(20)，扣减库存 | **`StockOut` + `ExecuteStockOutAsync`**，生成出库单并走库存三阶段逻辑 |

**结论（MVP）：**  
不必第一期就上「装箱单主表 + 拆分装箱 + 全套 EnumPackingStatus」。先用已有三张核心单据串起来：

```
出库通知 (StockOutRequest)
    → 生成拣货任务 (PickingTask) → 完成拣货
    → 执行出库 (StockOut)  // 扣库存、写出库单
    → （可选）登记物流单号 = 「出货」对外闭环
```

---

## 二、现有代码已具备的链路（可跑通前提）

1. **生成出库通知**  
   - 销售订单详情「申请出库」→ `POST /api/v1/stock-out/request` → `StockOutRequest`。

2. **出库通知列表**  
   - 页面：`StockOutNotifyList.vue` → 「执行出库」跳转执行页。

3. **执行出库页**（`StockOutEdit.vue`，路由约 `/inventory/stock-out/create?requestId=`）  
   - **生成拣货任务**：`InventoryCenter` → `GeneratePickingTask`（需带 `stockOutRequestId` 等）。  
   - **完成拣货**：`CompletePickingTask`，任务 `Status=100`。  
   - **执行出库**：`POST /api/v1/stock-out/execute` → `ExecuteStockOutAsync`（内部校验：**必须先有已完成拣货任务**）。

4. **库存与流水**  
   - 执行出库内含预占/拣货占用/出库确认逻辑，并调用 `RecordStockOutAsync`。

> **跑不通时的常见原因：** 未生成或未「完成」拣货任务即点「执行出库」——与后端校验一致，属预期。

## 三、简易流程定义（建议作为第一期产品流程）

### 3.1 角色（可简化）

| 步骤 | 建议操作角色 | 说明 |
|------|--------------|------|
| 出库通知 | 销售/商务 | 已具备 |
| 拣货（系统选批次） | 销售/商务（与文档一致：业务员侧可先选批次） | 与现有「生成/完成拣货任务」一致 |
| 备货（实物） | 仓库 | MVP **只做状态确认**，不做复杂质检 UI |
| 执行出库 | 仓管/有权限账号 | 扣账 |
| 出货（物流） | 仓管/物流 | 填写快递单号、发货时间 |

### 3.2 状态机（MVP：贴在「出库通知」或「拣货任务」上即可）

与《装箱拣货出库.md》**语义对齐**，数值可先不与 EBS 装箱单枚举混用，避免与现有 `StockOutRequest.Status(0/1/2)` 冲突：

| 阶段 | 对外展示名 | 建议实现方式（二选一） |
|------|------------|------------------------|
| A | 待拣货 | `StockOutRequest.Status=0` 且不存在 `Status=100` 的拣货任务 |
| B | 已拣货（对应文档 15） | 存在已完成拣货任务（`PickingTask.Status=100`） |
| C | 已备货（对应文档 16） | **MVP**：跳过；或增加 `StockOutRequest` 扩展字段 `ReadyAt` / `WorkflowSubStatus`，仅按钮「确认备货」写入，**不重复扣库存** |
| D | 已出库（对应文档完成出库） | `ExecuteStockOutAsync` 成功，`StockOutRequest.Status→1`，并生成 `StockOut` |
| E | 已发货（业务上「出货」） | **MVP**：在 `StockOut` 或扩展表上增加 `ExpressNo`、`ShippedAt`，或复用 `Remark` 临时承载 |

第一期若只关心「账实一致」，**D 即可视为业务闭环**；E 用于对接物流与客户通知。

## 四、第一期落地任务清单（按优先级）

### 4.1 必须（先跑通）— 第一期已实现

1. **导航串联**  
   - 出库通知列表 → 执行出库页：`requestId` 带入、自动拉取明细与仓库；执行页顶部 **四步流程说明**（Element 提示条）。  
   - 执行成功后跳转 **出库通知列表**（`StockOutNotifyList`）。

2. **校验与提示**  
   - 后端：`ExecuteStockOutAsync` 前校验通知未出库/未取消；成功后 **`StockOutRequest.Status = 1`**。  
   - 前端：未完成拣货时 **「执行出库」禁用**，`title` 提示原因；已出库通知 **禁用生成拣货**。

3. **出库通知状态回写**  
   - `StockOutService.ExecuteStockOutAsync` 在出库单落库并 `RecordStockOutAsync` 后，更新 **`StockOutRequest.Status = 1`**、`ModifyTime`。  

### 4.2 建议（闭环「出货」语义）

4. **发货信息**  
   - 在出库单或出库通知上增加可选字段：`快递单号`、`发货时间`、`承运商`；出库执行后由仓管补录或执行时一并填写。

5. **列表筛选**  
   - 出库通知列表：按「待拣货 / 已拣货未完成出库 / 已出库」筛选（由 `PickingTask` + `StockOutRequest.Status` 推导，无需用户理解内部 ID）。

### 4.3 暂缓（第二期再对齐截图）

- 独立 **装箱单主表**、客户装箱/拆分装箱、审核流、与文档完全一致的 **EnumPackingStatus 全路径**。  
- 与图片一致的 **装箱单列表** 大表、批量上传出库图等。

## 六、风险与假设

1. **库存数据**：执行出库依赖 `StockInfo` 可用量；测试环境需有对应物料、仓库、批次。目标架构下可用量由 **`stockitem` 聚合**后与 `StockInfo` 一致，见 `document/System/库存设计与实现方案-StockItem版.md`。  
2. **拣货任务与执行出库的物料行**：需保证 `ExecuteStockOutRequest.Items` 与申请数量、库存 FIFO 一致（现有服务已按仓库+物料扣减）；目标拣货绑定 **`StockItemId`** 时以该文档 §6 为准。  
3. **与 EBS 全量枚举对齐**：装箱单状态 **15/16/19/20** 建议在引入 **Packing** 实体时再落库；MVP 用「拣货任务 + 出库单」过渡，避免重复造两套状态机。
