# 重构建议

## 概述
本文件提供 FrontCRM 项目的架构级重构建议，旨在提升系统的可维护性、可扩展性、性能和安全性。

## 1. 架构重构

### 1.1 引入领域驱动设计（DDD）
**现状：** 当前采用传统分层架构，业务逻辑分散在服务层，领域概念不清晰。

**建议：**
1. **识别核心子域**
   - 认证与授权子域
   - 客户管理子域
   - 供应商管理子域
   - RFQ（需求）管理子域
   - 采购订单子域
   - 库存管理子域
   - 财务管理子域

2. **定义领域模型**
   - **聚合根：** Customer（客户）、Vendor（供应商）、RFQ（需求）、PurchaseOrder（采购订单）
   - **实体：** RFQItem（需求明细）、PurchaseOrderItem（采购订单明细）
   - **值对象：** Address（地址）、BankInfo（银行信息）、Contact（联系方式）
   - **领域服务：** DataPermissionService（数据权限服务）、SerialNumberService（流水号服务）

3. **实施领域事件**
   - `CustomerCreatedEvent`（客户创建事件）
   - `RFQSubmittedEvent`（需求提交事件）
   - `PurchaseOrderApprovedEvent`（采购订单审批通过事件）
   - 使用 MediatR 或类似库实现领域事件发布/订阅

### 1.2 实现 CQRS（命令查询职责分离）模式
**现状：** 同一服务类同时处理查询和命令，导致职责混杂。

**建议：**
1. **命令端（写操作）**
   - 使用 `ICommand` 和 `ICommandHandler<T>` 接口
   - 命令处理器负责业务逻辑、验证和持久化
   - 示例：`CreateCustomerCommand`、`UpdateRFQStatusCommand`

2. **查询端（读操作）**
   - 使用 `IQuery<T>` 和 `IQueryHandler<TQuery, TResult>` 接口
   - 查询处理器使用专门优化的查询模型
   - 可考虑使用 Dapper 或原生 SQL 提高查询性能
   - 示例：`GetCustomerListQuery`、`GetRFQDetailsQuery`

3. **分离数据存储（可选）**
   - 命令端：使用 Entity Framework Core 进行写操作
   - 查询端：使用只读副本数据库或视图

### 1.3 微服务拆分准备
**现状：** 单体架构，所有功能部署在一个进程中。

**建议：**
1. **识别服务边界**
   - **认证服务 (AuthService)：** 用户认证、权限管理
   - **客户服务 (CustomerService)：** 客户信息管理
   - **供应商服务 (VendorService)：** 供应商信息管理
   - **RFQ服务 (RFQService)：** 需求管理
   - **采购服务 (PurchaseService)：** 采购订单管理
   - **库存服务 (InventoryService)：** 库存管理
   - **财务服务 (FinanceService)：** 财务管理

2. **定义服务接口**
   - 使用 gRPC 或 REST API 作为服务间通信协议
   - 定义清晰的 API 契约和版本策略

3. **数据所有权划分**
   - 每个服务拥有自己的数据库
   - 使用 Saga 模式处理跨服务事务

## 2. 数据库优化

### 2.1 数据库设计优化
**建议：**
1. **索引策略**
   - 为所有外键字段添加索引
   - 为常用查询字段（如 `SalesUserId`、`PurchaseUserId`、`Status`）添加索引
   - 考虑复合索引优化联合查询

2. **主键优化**
   - 考虑使用 `bigint` 序列代替 GUID 作为主键
   - 或使用组合键（时间戳+序列）减少索引碎片

3. **分区策略**
   - 按时间分区日志表（如 `SysOperationLog`）
   - 按业务模块分区大表

### 2.2 引入数据库迁移工具
**建议：**
1. **使用 DbUp 或 FluentMigrator**
   - 纯 SQL 迁移，更可控
   - 支持回滚脚本
   - 与 CI/CD 流水线集成

2. **迁移策略**
   - 开发环境：自动应用迁移
   - 测试环境：手动审核后应用
   - 生产环境：DBA 审核后手工执行

## 3. 安全架构加固

### 3.1 认证授权增强
**建议：**
1. **OAuth 2.0 / OpenID Connect 集成**
   - 支持第三方身份提供商（如 Azure AD、企业微信）
   - 实现标准的 OIDC 流程

2. **多因素认证（MFA）**
   - 支持 TOTP（时间基一次性密码）
   - 支持短信/邮件验证码

3. **令牌管理**
   - JWT 刷新令牌机制
   - 令牌吊销列表（可选）
   - 短期访问令牌（15分钟）+ 长期刷新令牌（7天）

### 3.2 网络安全
**建议：**
1. **API 安全**
   - 实施 API 密钥认证（用于服务间调用）
   - 添加 API 速率限制（使用 AspNetCore.RateLimiting）
   - 请求签名验证（防止重放攻击）

2. **安全标头**
   - Content Security Policy (CSP)
   - HTTP Strict Transport Security (HSTS)
   - X-Frame-Options, X-Content-Type-Options

### 3.3 数据安全
**建议：**
1. **数据加密**
   - 使用 .NET 数据保护 API 加密敏感字段
   - 数据库字段级加密（使用 PostgreSQL pgcrypto）

2. **审计日志**
   - 完善的操作日志记录
   - 数据变更历史（CDC 或触发器）

## 4. 性能优化架构

### 4.1 缓存策略
**建议：**
1. **多级缓存架构**
   - **L1 缓存（内存）：** 使用 IMemoryCache 缓存用户权限、字典数据
   - **L2 缓存（分布式）：** 使用 Redis 缓存共享数据、会话状态
   - **L3 缓存（HTTP缓存）：** 使用响应缓存中间件缓存静态数据

2. **缓存策略**
   - 权限数据：5分钟过期，滑动过期
   - 字典数据：30分钟过期，绝对过期
   - 列表数据：根据变更频率设置过期时间

### 4.2 异步处理
**建议：**
1. **消息队列集成**
   - 使用 RabbitMQ 或 Azure Service Bus
   - 异步处理：邮件发送、报表生成、数据同步

2. **后台任务**
   - 使用 Hangfire 或 Quartz.NET
   - 定时任务：数据清理、统计计算、缓存预热

### 4.3 前端性能优化
**建议：**
1. **应用现代化**
   - Vue 3 Composition API 全面迁移
   - 使用 `<script setup>` 语法糖
   - 引入 VueUse 工具库

2. **构建优化**
   - Vite 配置优化（分包策略、代码压缩）
   - 图片资源优化（WebP 格式、懒加载）
   - 字体优化（子集化、预加载）

## 5. 可观测性架构

### 5.1 监控告警
**建议：**
1. **应用性能监控（APM）**
   - 集成 Application Insights 或 OpenTelemetry
   - 跟踪请求链路、数据库查询、外部调用

2. **日志聚合**
   - 使用 Serilog + Seq/ELK Stack
   - 结构化日志，便于查询分析

3. **健康检查**
   - 实现健康检查端点（/health）
   - 检查数据库连接、缓存连接、外部服务

### 5.2 告警机制
**建议：**
1. **关键指标监控**
   - 错误率 > 1%
   - 响应时间 P95 > 2秒
   - 数据库连接池使用率 > 80%

2. **告警渠道**
   - 企业微信/钉钉机器人
   - 邮件通知
   - SMS（关键告警）

## 6. 部署与运维架构

### 6.1 容器化优化
**建议：**
1. **Docker 镜像优化**
   - 多阶段构建减少镜像大小
   - 使用 Alpine Linux 基础镜像
   - 非 root 用户运行容器

2. **Kubernetes 部署**
   - 定义 Deployment、Service、Ingress
   - 配置 Horizontal Pod Autoscaler（HPA）
   - 使用 ConfigMap 和 Secret 管理配置

### 6.2 数据库高可用
**建议：**
1. **PostgreSQL 高可用**
   - 主从复制（流复制）
   - 自动故障转移（Patroni、Stolon）
   - 连接池（PgBouncer）

2. **备份策略**
   - 每日全量备份 + 每小时增量备份
   - 异地备份（云存储）
   - 定期恢复测试

## 7. 开发流程改进

### 7.1 测试策略
**建议：**
1. **测试金字塔**
   - 单元测试（覆盖率 > 70%）：使用 xUnit/NUnit
   - 集成测试：使用 Testcontainers 测试数据库交互
   - 端到端测试：使用 Cypress/Playwright

2. **测试数据管理**
   - 使用测试数据工厂
   - 数据库快照恢复
   - 模拟外部服务

### 7.2 CI/CD 流水线
**建议：**
1. **持续集成**
   - 代码质量检查（SonarQube）
   - 自动化测试执行
   - 安全扫描（SAST）

2. **持续部署**
   - 蓝绿部署或金丝雀发布
   - 自动化回滚机制
   - 环境一致性验证

## 实施路线图

### 第一阶段：基础加固（1-2个月）
1. 修复高优先级安全问题
2. 添加关键数据库索引
3. 统一异常处理和日志
4. 实现缓存基础架构

### 第二阶段：架构优化（3-4个月）
1. 引入 DDD 概念，重构核心领域
2. 实现 CQRS 模式
3. 数据库优化和迁移工具
4. 前端性能优化

### 第三阶段：高级特性（5-6个月）
1. 微服务拆分准备
2. 高级安全特性（OIDC、MFA）
3. 完整的可观测性体系
4. 自动化运维平台

### 第四阶段：规模化（7-12个月）
1. 微服务实施
2. 多租户支持
3. 国际化/本地化
4. 移动端适配

## 成功度量指标
- 系统可用性：99.9%
- API 平均响应时间：< 500ms
- 数据库查询 P95：< 100ms
- 单元测试覆盖率：> 70%
- 部署频率：每日多次
- 变更失败率：< 5%

## 风险与缓解
1. **架构重构风险**：逐步重构，保持向后兼容
2. **数据库迁移风险**：充分测试，分阶段迁移
3. **团队技能风险**：培训、知识分享、外部专家支持
4. **时间成本风险**：分阶段实施，优先高价值改进

---

**结论：** FrontCRM 项目具备良好的基础，通过上述架构重构建议，可逐步演变为高性能、高可用、易维护的现代化企业应用系统。建议采用渐进式重构策略，优先解决影响业务连续性和安全性的问题。
